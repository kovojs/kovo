# Optimistic updates

Optimism in Jiso is keyed to **queries** (the data), never to islands (the UI). You declare one
transform per (mutation × invalidated query); every island consuming that query updates from it —
including islands written months after the mutation (SPEC §10.4). There is no per-component
optimistic state to wire and no cache to patch.

## Declare transforms with OptimisticFor

A transform is a pure `(current, input)` function over the query's result type. From the commerce
reference app:

```ts
import { form } from '@jiso/core';
import type { OptimisticFor } from '@jiso/runtime';

export const addToCartForm = form<'cart/add', AddToCartInput>('cart/add');

export const addToCartOptimistic = {
  queue: 'cart',
  transforms: {
    cart(current, input) {
      return { count: (current?.count ?? 0) + input.quantity };
    },
    orderHistory: 'await-fragment',
    productGrid: 'await-fragment',
  },
} satisfies OptimisticFor<typeof addToCartForm>;
```

Three things to notice:

- **The keys are query names**, and the set of required keys is the mutation's _derived_
  invalidation set — the compiler emits each mutation's invalidated-query keys into the registries,
  so `OptimisticFor<typeof addToCartForm>` demands an entry per invalidated query in plain `tsc`
  (SPEC §10.6, §6.1). Add a write that touches a new domain and this object goes red.
- **`'await-fragment'` is a first-class answer.** It documents "considered; the 1-RTT latency is
  accepted here" — the product grid re-renders from the server fragment instead of being predicted.
  Deliberate deferral and forgotten coverage are different states, and only one of them is a
  diagnostic (SPEC §10.4, §10.6).
- **`queue: 'cart'`** serializes this mutation's submissions through a named FIFO when ordering
  matters (SPEC §10.4).

Transforms are written against the query's inferred result type, so a column rename breaks the
transform in the editor, not in production.

## Why UNHANDLED fails fw check

Coverage = invalidated-query set × status, checked per mutation. The valid v1 statuses are
`hand-written` and `await-fragment`; anything else is FW310 (SPEC §10.6). The check runs at two
altitudes off the same derived set: as a type error in the editor (via the emitted invalidation
sets) and as `fw check` in CI.

Here is the failure mode, generated by running `fw check` against the commerce reference graph
with the hand-written `cart` transform deleted:

```txt
fw-check/v1
WARN FW310 cart/add -> cart Invalidated query lacks optimistic transform.
```

`fw check` exits non-zero on FW310, so the gap cannot ship silently. The corresponding
`fw explain mutation cart/add --optimistic` run shows the same gap inline with its fix menu:

```txt
OPTIMISTIC productGrid await-fragment
OPTIMISTIC orderHistory await-fragment
OPTIMISTIC cart UNHANDLED
  -> hand-write in the mutation module, or declare 'await-fragment'
OPTIMISTIC-SUMMARY total=3 hand-written=0 await-fragment=2 UNHANDLED=1
```

With the transform in place (the commerce app as committed), the same command reports:

```sh
fw explain mutation cart/add --optimistic graph.json
```

```txt
fw-explain/v1
MUTATION cart/add
guards: authed,rateLimit:session
session: commerceSession
input-fields: productId,quantity
writes: cart,product,order
invalidates: cart,product,order
manual-invalidates: -
updates: cart->component:CartBadge,page:/cart; orderHistory->component:OrderHistory,page:/cart; productGrid->component:ProductGrid,page:/cart
OPTIMISTIC cart hand-written
OPTIMISTIC productGrid await-fragment
OPTIMISTIC orderHistory await-fragment
OPTIMISTIC-SUMMARY total=3 hand-written=1 await-fragment=2 UNHANDLED=0
```

Forgetting an optimistic update is a visible, suppressible diagnostic with the suppression
recorded in source — never a silent UI inconsistency (SPEC §10.6). This is the same theorem the
framework applies one hop further down: every query-dependent DOM position needs a declared
update status too (FW311, SPEC §4.9).

## The runtime protocol

When the user submits, the loader runs a fixed sequence (SPEC §10.4):

1. **Snapshot** the affected query values (`structuredClone` — safe because query data is
   `JsonValue` by construction).
2. **Apply transforms** to the shared query values and run their update plans. Every dependent
   island updates at once.
3. **Stamp pending state.** Affected islands get `fw-pending` and `aria-busy="true"`
   automatically — style the in-flight state with CSS, no per-component spinner wiring:

```html
<cart-badge fw-deps="cart" fw-pending aria-busy="true">…</cart-badge>
```

4. **On success**, the response's `<fw-query>` values and fragments reconcile over the prediction.
   A right guess is a near-no-op morph; a wrong guess is silently corrected. Server truth always
   wins — predictions are throwaway sketches, not a consistency protocol (SPEC §2, §10.4).
5. **On error**, the snapshots are restored and the typed error fragment renders
   (see [the 422 path](/guides/mutations/)).

The client treats missing server truth for an applied transform as a visible runtime diagnostic,
then settles that transform without promoting the prediction to authoritative data — explicitly
fragment-only responses are allowed, silent inconsistency is not (SPEC §10.4).

## Rebase: concurrent mutations

Each query keeps a pending-transform log. When server truth arrives while other mutations are
still in flight, the runtime morphs the authoritative value in, then re-applies the still-pending
transforms in order — a rebase, safe because transforms are pure `(data, input)` functions
(SPEC §10.4).

```ts
// what the loader does, in @jiso/runtime terms
import { OptimisticRebaser, createQueryStore } from '@jiso/runtime';

const store = createQueryStore();
const rebaser = new OptimisticRebaser(store);

rebaser.add('m1', { productId: 'p1', quantity: 2 }, addToCartOptimistic); // predict
rebaser.applyServerTruth('cart', { count: 7 }); // morph truth in, re-apply pending
rebaser.settle('m1'); // m1's response landed
```

Navigation is a free reconciliation point: in-flight mutations complete via `keepalive`, and the
pending log dies with the document — stale optimism outliving its mutation is structurally
impossible (SPEC §8, §10.4).

## Test the prediction

A transform is a pure function, so unit-test it directly — and property-test that the prediction
is contained in eventual truth over generated states (SPEC §12, §11.4):

```ts
import { propertyTest } from '@jiso/test';

expect(addToCartOptimistic.transforms.cart({ count: 1 }, { productId: 'p1', quantity: 2 })).toEqual(
  { count: 3 },
);

// prediction ⊆ eventual truth: patch(shape(s), input) ≡ shape(apply(effect, s, input))
propertyTest({
  apply: (state, input) => applyAddToCartEffect(state, input), // server-side effect
  shape: (state) => shapeCartQuery(state), // what the query ships
  predict: (state, input) => addToCartOptimistic.transforms.cart(shapeCartQuery(state), input),
  cases: generatedCartStates(),
});
```

See the [testing guide](/guides/testing/) for the harness this runs in.

## Hand-written now, derived later

v1 transforms are hand-written against the same transform IR that v2's compiler derivation will
emit. For writes whose dataflow is closed over the mutation input, schema constants, and data the
query already ships, v2 generates the transform — and because the IR is shared, adoption is
incremental: delete a hand-written transform and derivation takes over, pair by pair. Underivable
cases punt loudly, with the exact expression and reason named in `fw explain --optimistic`
(SPEC §10.4, §10.5).

## When _not_ to predict

`'await-fragment'` is the right answer more often than SPA habits suggest:

- The server computes something the client cannot (totals with tax rules, ranking, inventory
  races) — a wrong prediction is worse than a 1-RTT wait (SPEC §10.5's punt philosophy).
- The query feeds a `fragment`-status region — fragment positions get no optimistic update by
  definition; the server re-render is the update (SPEC §4.9).
- The data is rarely on screen during the mutation.

Declare the deferral and move on; the check is satisfied either way.
