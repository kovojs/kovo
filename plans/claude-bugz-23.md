## Issues

### A. Query-loader read-set erasure → owner-scope IDOR + secret-to-wire fail open

- [ ] **B1 — An inline `const db = context.db` (or `context!.db!`) binding in a query loader erases the static read set, so KV414 (owner-scope IDOR) and KV435 (secret-to-wire) both fail OPEN on a green `kovo build`.** (HIGH, framework/security; found by `auth-access`, self-verified by isolation flip)
  - Observed: a query loader reading an owner-scoped `notes` table by client `id` with the obvious inline handle builds GREEN; `graph.json` records the query with `domains: []` and `kovo explain --unscoped` reports `SUMMARY total=0` (false all-clear). The identical erasure defeats KV435: a query projecting the `secret`-classified `account.password` via the inline handle builds GREEN and would ship the scrypt hash on `/_q`.
  - Root cause: `packages/server/src/query.ts:55-58` types `QueryLoadContext.db?: Reader<Db>` **optional**, so every natural inline binding is `Reader<Db> | undefined`. `packages/drizzle/src/static/schema.ts:296-310` (`isKovoReaderOfDrizzleDatabaseType`) recognizes the read handle **only** when the receiver type's alias symbol is literally `Reader` (`type.getAliasSymbol()?.getName() !== 'Reader' → return false`). A `Reader<Db>|undefined` union, or a type produced by a `NonNullExpression` `!`, loses that top-level alias symbol, and `Reader<Db>` is a write-stripped mirror so the structural fallback in `isDrizzleDatabaseType` does not rescue it. The read-extraction then emits **zero reads** (fail OPEN) instead of degrading to KV406 manual-touches/un-analyzable (fail CLOSED); with an empty read set KV414/KV435/KV407 cannot fire. Only an explicit `Reader<Db>` annotation (the starter's `requireDb(context): Reader<AppDb>` helper) preserves the alias and re-enables the gates.
  - Why it matters: cross-tenant read (IDOR) AND credential-hash disclosure both ship through a green build with a false `kovo explain --unscoped` all-clear. The starter is safe only by accident (its `requireDb` helper carries a `Reader<AppDb>` return annotation); the obvious inline `const db = context.db` that any app author writes is silently unprotected. An un-provable read receiver must fail CLOSED (KV406/KV433), not emit zero reads.
  - Repro evidence (SELF-VERIFIED, isolation flip): owner-scoped `notes` (`kovo({domain,key:t=>t.id,owner:t=>t.userId})`) + a query `load` doing `const db = context!.db!; db.select(...).from(notes).where(eq(notes.id,id))`. **Build A (inline):** `build:prod` exit 0, GREEN, no KV414. **Build B (one-token change to `const db: Reader<AppDb> = context!.db!`, a runtime no-op):** `build:prod` exit 1 → `ERROR KV414 QUERY probe/note-by-id-query domain=model/note key=arg:id scope=args ... (IDOR).` + `KV407`. The static type is the _only_ difference.
  - Acceptance: a query loader whose `context.db` receiver cannot be statically proven a `Reader<Db>` fails CLOSED (KV406 "un-analyzable read; declare reads" / KV433), never emitting an empty read set; OR `QueryLoadContext.db` is non-optional and the recognizer follows `!`/union narrowing. A focused test: the inline `const db = context.db` owner-scoped-by-arg query must fail KV414. (Distinct from `papercuts-2`, which fixed only the `requireDb`-helper functional-fragment case; this is the read-side security escalation of the same extraction miss.)

### B. Silent SSR truncation of the primary read

- [ ] **B2 — The SSR document silently truncates a query list at the 100-item API4 ceiling, discarding the `QUERY_LIST_LIMIT` warning that the `/_q` path emits — the no-JS primary read shows incomplete data with a misleading count and no signal.** (MED, framework/soundness; found by `typed-reads`, verified independently)
  - Observed: with 153 contacts, the prod SSR document (`GET /`) renders only the first 100 rows, the count badge reads "100" (true total 153), and the document carries **no** `kovo-warn` header. The same query over `/_q/queries/contacts-query` returns 100 items **with** `kovo-warn: QUERY_LIST_LIMIT $.items;limit=100`; a sibling `count()` aggregate truthfully reports 153 — so the loader returns all rows and 53 are dropped post-load on the render path with no signal.
  - Root cause: the cap `DEFAULT_MAX_QUERY_LIST_ITEMS=100` (`packages/server/src/app-load-shed.ts:43-44`) is correct (SPEC §9.5 API4). `capQueryListResults` truncates and records `QUERY_LIST_LIMIT` warnings (`packages/server/src/query.ts:836-864`); `runQuery` returns `{value, warnings}`. The `/_q` path surfaces them via `queryWarningHeaders` (`query.ts:701,970-979`), but the SSR render path threads `maxListItems` (`packages/server/src/app-document.ts:103`) into `packages/server/src/jsx-runtime.ts:681`, which assigns **only** `result.value` and **discards `result.warnings`** — so the document emits no `Kovo-Warn` and the component cannot detect the cap.
  - Why it matters: the headline read silently hides data past 100 rows on the no-JS path (the only users who ever see the SSR document) with neither an operator header nor a component-detectable signal — fail-silent incomplete data. The /\_q-vs-SSR asymmetry shows the framework chose to signal truncation, then drops the signal on the primary path.
  - Repro evidence: seed >100 contacts → `GET /` badge "100", highest card "Person 097", no `kovo-warn`; `GET /_q/queries/contacts-query` → 100 items + `kovo-warn` header; `GET /_q/<count-probe>` → `total:153`. (Honest bound: the count badge renders `items.length` — partly app-authored; the cap itself is correct and data is not corrupted.)
  - Acceptance: the SSR render path threads `result.warnings` into the document (emit `Kovo-Warn` and/or make the truncation component-detectable), so a capped primary read is never silent; consider a starter pagination affordance.

### C. File type-enforcement fail-open on the validate-only path

- [ ] **B3 — `s.file().accept([...])` WITHOUT `.store()` trusts the client-declared MIME instead of the server-sniffed bytes — the documented server-truth allowlist is bypassable on the validate-only path.** (MED, framework/security; found by `files-capability`, verified independently; residual of the `[x]`-fixed `.store()` enforcement)
  - Observed (public-API repro): a file of HTML bytes `<html><script>alert(document.cookie)</script></html>` with client `type:'image/png'` — `s.file().accept(['image/png']).parse(lyingFile)` is **ACCEPTED** (returns `type=image/png`, no sniff); `s.file().accept(['image/png']).store({storage}).parseAsync(lyingFile)` is **REJECTED** (server-sniffed). Opposite enforcement for the same allowlist depending on whether `.store()` is chained.
  - Root cause: `packages/server/src/schema.ts:882-898` (`parseFileLike`) checks the `.accept(...)` allowlist against `input.type` (client MIME) on the sync parse path (comment: "a cheap client-MIME pre-filter only"). The authoritative `sniffUploadBytes` lives ONLY in `StoredFileSchemaImpl.parseAsync` (`schema.ts:769-793`). A bare `s.file().accept([...])` `FileSchema` is not an `AsyncSchema`, so the mutation pipeline's `parseSchemaAsync` (`schema.ts:1043-1052`) dispatches to sync parse and never sniffs.
  - Why it matters: the public JSDoc on `FileSchema.accept` (`schema.ts:335-340,348-352`) promises the plain `accept([...])` form checks the **server-sniffed bytes**, with no ".store() only" caveat (KV428 exists to kill the verbatim-client-MIME hole). An author who validates an upload then processes/serves the bytes itself (resize, hash, forward, serve inline) — without `.store()` — gets only the client MIME lie; serving those bytes inline is stored XSS. The `.store()` path (prior `[x]`-fixed item) still correctly rejects — this is the non-store residual.
  - Repro evidence: public-API vitest above (2/2 assertions); source dispatch `parseSchemaAsync` (`schema.ts:1043-1052`) → sync `parseFileLike` for a non-store `FileSchema`.
  - Acceptance: `s.file().accept([...])` enforces the server-sniffed type on every path (make the bare `FileSchema` async-sniffing, or fail closed / diagnose when an `accept` allowlist is used without a sniff), matching its JSDoc; or the JSDoc is corrected and a build diagnostic steers authors to `.store()`/explicit sniff.

### D. Webhook write-audit under-reports actual writes

- [ ] **B4 — A webhook `recordChange()` to a domain NOT in its declared `writes` is silently permitted and omitted from `kovo explain --endpoints`, yet emitted on the wire — the §11.4 machine-ingress write audit under-reports what a webhook can write/invalidate.** (MED, framework/soundness; found by `endpoints-webhooks`, verified independently; residual of the `[x]`-fixed `bugz-9` B2)
  - Observed: a webhook declaring `writes: [contact]` whose handler also calls `ctx.recordChange(billing, ...)` (where `billing ∉ writes`) builds clean; at runtime the response `kovo-changes` header includes `{"domain":"billing",...}`, but `kovo explain --endpoints` prints `writes=model/contact` only — `billing` is on the wire but absent from the audit.
  - Root cause: `packages/server/src/webhook.ts:751-761` `recordChange` pushes any `domain` onto the change set with no check that `domain.key ∈ writes`; the audit's write column is sourced solely from the static `writes` array, and nothing reconciles runtime `recordChange` against it. The type signature (`webhook.ts:92`) accepts any `Domain` with no declared-writes union, so there is no type-level guardrail either (contra CLAUDE.md type-level-security doctrine). Mutations fail CLOSED (KV406/KV414) on an undeclared write; the webhook path permits the identical divergence silently.
  - Why it matters: `bugz-9` B2 made the static `writes` list the audit's source of truth, but the runtime never reconciles `recordChange` against it, so a security reviewer auditing "what can this webhook touch" via `kovo explain` misses `billing`. Asymmetric audit-honesty gap on the machine-ingress write chain (SPEC §11.4).
  - Repro evidence: build green; `kovo explain --endpoints` shows `writes=model/contact`; a valid HMAC-signed POST returns `kovo-changes: [{"domain":"model/contact",...},{"domain":"billing",...}]`. (Honest bound: webhook writes are declaration-based by SPEC design and the wire is fully honest — only the static audit under-reports, and only when an author misdeclares.)
  - Acceptance: constrain `recordChange`'s `domain` to the declared-`writes` union (type-level) and/or fail closed / diagnose when `recordChange` targets a domain outside `writes`, so the §11.4 audit cannot under-report.
