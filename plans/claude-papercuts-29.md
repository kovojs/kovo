# Round-10 Papercuts 29

Created 2026-07-02. Source of truth remains `SPEC.md`. Over-block + dialect-parity + ergonomics items from the
Round-10 paranoid dogfood AFTER `plans/fundamental-fixes-followup-5.md` (authorization runtime chokes) landed.
Security fail-opens are in `plans/claude-bugz-31.md`.

**Meta-theme — the SQLite authorization choke fails the SAME way in BOTH directions, the classic incomplete-classifier
signature, now on writes and non-flat reads.** `bugz-31` is the fail-OPEN half (owner reads escaping the proxy); this
ledger is the fail-CLOSED half: the SQLite choke categorically denies legitimate owner-self operations that Postgres's
`WITH CHECK`/RLS admits — owner INSERTs (P1) and subquery-FROM reads (P2) — plus the durable-task principal seam is
never consumed (P3) and the SQLite runtime-db artifact is hand-maintained where Postgres derives it (P4). The Postgres
dialect is the reference for correct behavior in every one of these; SQLite diverges because its enforcement is a
builder-shape proxy rather than a connection/statement-level door.

## Issues

- [ ] **P1 — SQLite owner-scoped INSERT is unconditionally rejected at runtime (KV414): a legit owner cannot create their own row. Build is green, the request 500s, and the identical app works on Postgres — a hard dialect-parity break on the canonical "user creates their own owned row" pattern.** (HIGH-severity papercut, over-block, architectural; converged from `idor-core-both-dialects` + `overblock-regression-realistic`; reproduced, paranoid-confirmed)
  - Observed: `notes`/`orders` = `kovo({owner:'userId'})`. A guarded, CSRF-protected `createNote`/`addOrder` mutation whose owner column is set from `request.session.user.id` (the exact blessed owner-insert shape, matching `examples/commerce/src/domain.ts:256-265`) builds GREEN under `KOVO_PARANOID=1` but returns **HTTP 500** at runtime with server log `KV414: SQLite managed insert into an owner-scoped table requires owner-checkable framework proof (SPEC §10.3)`. No row is persisted. Reproduced for multiple users. The SAME source on `--dialect postgres` returns 303 and inserts succeed (RLS `WITH CHECK(owner=principal)` admits owner-self inserts); each user sees only their own rows, no cross-owner leak. Owner UPDATE/DELETE of one's OWN rows work (`changes=1`) — the block is INSERT-specific. The default scaffold `contacts` table (`authzPolicy`, shared) still inserts fine, so the census did not break the out-of-box template; the break hits any `kovo({owner})` table.
  - Root cause (self-verified): `packages/server/src/managed-db.ts:1059` `assertSqliteInsertIsOwnerCheckable()` — for any table classified `owned`/`ownedVia` it UNCONDITIONALLY `throw new Error('KV414: …')` at `:1073-1075`. **No branch inspects `config.values` to admit an owner column bound to the principal** (contrast the update path `assertSqliteUpdateDoesNotReassignOwner` at `:1078`, which allows). Reached from `applySqliteAuthorizationToBuilder()` insert branch (`:915-917`), fired on `.values()` (`:879-881`). The generated write handle composes it: `src/_kovo/app-runtime-db.ts:165 createSqliteAuthorizationDb(createDeclaredWriteDb(…))`, so `request.db.insert(ownerTable).values({…})` in every mutation throws before SQL executes — regardless of `serverValue`, the owner column, or a discharged static KV414 write gate. SQLite has no storage-engine analogue to Postgres `WITH CHECK`, and followup-5 added no request-path owner-checkable insert (its 1.2 says SQLite "writes fail closed when un-owner-checkable" — but "un-owner-checkable" was implemented as "all owner inserts"). `SPEC.md §10.3` defines NO "owner-checkable insert" requirement; this is a followup-5 implementation artifact, not a documented non-goal.
  - Why it matters: **the most basic owned-data operation — a user creating their own row — is impossible on the SQLite starter, with a confusing green-build→HTTP-500 and no supported alternative** (`declarePublicRead`/`rawRead` are read-only; `actAs`/`declareSystem` are task/webhook-only). An app that works on Postgres is categorically broken on SQLite for every owner table. The dual of `bugz-31` P-open: the proxy over-denies writes exactly as it under-scopes reads.
  - Repro evidence: `/Users/mini/kovo-dogfood-round10/{notes-sqlite,vr-owninsert,idor10sq}` — SQLite `createNote`/`addOrder` → 500 with the KV414 owner-insert string; `add-contact` (non-owned) → 303. `/Users/mini/kovo-dogfood-round10/notes-pg` — identical source, `createNote` → 303, inserts succeed, per-owner isolation intact. First-party test `sqlite-authz.test.ts:161-163` already asserts even a correct `db.insert(orders).values({…,userId:'u1'})` with `principal:'u1'` throws `/insert into an owner-scoped table/`. Self-verified: `managed-db.ts:1059-1076`.
  - Acceptance: the SQLite managed insert into an owner-scoped table SUCCEEDS when the owner column is bound to the request principal (mirror Postgres `WITH CHECK(owner=principal)`): inspect `config.values`, require the owner column === principal, admit it, and fail closed only when the owner column is absent/mismatched/principal-less. A paranoid test: an owner creating their own row on SQLite persists; setting the owner column to a different principal is rejected. This is the write dual of `bugz-31` — box exactly the disallowed writes, no more.

- [ ] **P2 — Subquery-in-FROM over an owned table is denied wholesale (`1=0`) instead of scoped, so the legitimate owner cannot read their OWN rows through a subquery.** (LOW, over-block, architectural; `sqlite-injection-version-split`; reproduced, paranoid-confirmed)
  - Observed: signed in as Alice, `GET /_q/queries/orders-sub-from-query` (`SELECT … FROM (SELECT … FROM orders) sub`) returns `{items:[]}` — Alice's own row is missing, even though the plain builder read returns it. Fail-CLOSED (no cross-owner leak), but breaks a legitimate read pattern.
  - Root cause: `packages/server/src/managed-db.ts` `collectSqliteAuthorizationPredicateForTable()` (`:1000-1008`) — when the FROM target is a subquery, `sqlitePhysicalTableName()` returns undefined, so it delegates to `sqliteSubqueryUsesDeniedTable()` (`:1041-1057`), which returns true whenever the subquery's `_.usedTables` includes any `owned`/`ownedVia`/`authzPolicy` table, pushing `sql`1 = 0`` onto the OUTER select's predicates (`:1003`). It denies the whole read rather than recursing into the subquery builder to inject the inner owner predicate.
  - Why it matters: a standard SQL shape (`FROM (subquery)`) over one's own owned table always returns zero rows. Same recursion gap that `bugz-31` B2 exposes on the relational builder — the predicate injector handles flat `.from(table)` but not nested/compound read shapes.
  - Repro evidence: `/Users/mini/kovo-dogfood-round10/vsplit-app`; `orders-sub-from-query` as Alice → `{items:[]}` while `orders-plain-query` as Alice → her row. Root at `managed-db.ts:1000-1008`, `:1041-1057`.
  - Acceptance: recurse into the subquery builder and inject the inner owner predicate (scope, don't deny), so an owner reading from a subquery of their own owned table sees their rows; deny only when no arm can be owner-scoped.

- [ ] **P3 — The durable-task ingress builds an `actAs`/`declareSystem` principal posture but the query/mutation runners never consume it, so a task's `ctx.actAs(u)` read/write of an owner table denies (zero rows / no-op) instead of scoping to `u`.** (MED, over-block, architectural; `ingress-endpoints-tasks-webhooks`; CONFIRMED by static chain, fail-closed)
  - Observed: a durable task's idiomatic `ctx.actAs(u).runQuery/runMutation` against owner tables returns zero rows / no-op writes instead of `u`'s data; `declareSystemRead/Write` reads nothing. Fail-CLOSED (no cross-owner leak), but the actAs seam followup-5 added for tasks/webhooks is inert on the task path.
  - Root cause (static chain verified end-to-end): `packages/server/src/task-runtime.ts:107,130` pass `principalPosture: ingressOptions.principalPosture` into `runMutation()`/`runQuery()` options. But `packages/server/src/query.ts` `runQuery` (`:418-424`) forwards only `clientIp/db/onError/sessionProvider` to `resolveKovoLifecycleRequest` and never reads `options.principalPosture`; `packages/server/src/mutation.ts` has zero `principalPosture` references. Only `endpoint.ts:290-313` attaches the posture onto the request (via `Object.defineProperty`) BEFORE calling `options.db(dbRequest)`, which is what `appRuntimeDbProvider`'s `ownerPrincipalFromRequest` inspects (`app-runtime-db.sqlite.ts:185-219`). The task path attaches it to neither the request nor the lifecycle options, and `task-runtime.ts:215-217` builds a bare `new Request(url,{method:'POST'})` (no cookies → `sessionProvider` yields null), so `appRuntimeDbProvider(taskRequest)` gets `principal=undefined` → `managed-db.ts:1024` returns `sql`1 = 0`` for owner tables.
  - Why it matters: followup-5's `actAs`/`declareSystem` principal seam — the whole mechanism for tasks/webhooks to act on owner data — silently no-ops on the durable-task surface; task-owned reads/writes fail closed with no diagnostic. (Endpoints work: `endpoint.ts:290` threads it correctly, verified in `bugz-31` B1's `actas-read`.)
  - Repro evidence: source chain `task-runtime.ts:107,130` → `query.ts:418-424` (unused) → `mutation.ts` (absent) → `endpoint.ts:290-313` (the only consumer) → `app-runtime-db.sqlite.ts:185-219` (principal only from `request.principalPosture`/`request.session.user.id`, neither present on a task request). Full-harness serve not stood up because every link is verified and the `1=0` outcome is deterministic.
  - Acceptance: `runQuery`/`runMutation` thread `options.principalPosture` into the lifecycle request (attach it the way `endpoint.ts` does) so `appRuntimeDbProvider` derives the task principal; a task's `ctx.actAs(u)` scopes owner reads/writes to `u`, and `declareSystem` reads the declared system scope. Add a task-path test asserting an owner read under `actAs` returns that owner's rows.

- [ ] **P4 — The SQLite `src/_kovo/app-runtime-db.ts` is a one-time static scaffold artifact (literal `SCHEMA_DDL` + hand-maintained `SCHEMA_TABLES`), not regenerated by build, so adding an owner table requires hand-writing CREATE TABLE DDL + registering it — where Postgres derives both from `schema.ts`.** (LOW, ergonomics/dialect-parity; `overblock-regression-realistic` + `pg-rls-confinement-escalation`; self-verified)
  - Observed: on the `--sqlite` scaffold, `src/_kovo/app-runtime-db.ts` hardcodes `SCHEMA_DDL` as literal `CREATE TABLE` strings and a hand-maintained `SCHEMA_TABLES` array fed to `extractKovoRuntimeDbMetadata`. `kovo build` does not regenerate it. Adding a `notes` table to `schema.ts` has no runtime effect unless the author also manually adds the DDL AND registers the table in `SCHEMA_TABLES`; otherwise the table doesn't exist at runtime and/or is unclassified. The `--dialect postgres` scaffold instead computes `SCHEMA_DDL = schemaDdl(SCHEMA_TABLES)` and applies RLS via `applyPgliteOwnerPolicies`, so only `SCHEMA_TABLES` needs the entry (DDL auto-derived).
  - Root cause: `packages/create-kovo/templates/src/_kovo/app-runtime-db.sqlite.ts` ships 5 literal `CREATE TABLE` strings and zero `schemaDdl()` calls; the PG template computes DDL at module load. `create-kovo` copies the file once; no build/codegen re-derives it (grep of `packages/` + `scripts/` shows only the scaffold copy references it). Compounding: `runtime-metadata.ts:129-138` seeds census `schemaTableNames`/`ownerSourcesByTable` only from the passed `SCHEMA_TABLES`, which diverges from `schema.ts` (the static KV414 gate reads `schema.ts` directly), and `managed-db.ts:665 if (!inSchema) continue;` skips a table absent from that metadata (fail-safe only because such a table also gets no DDL — see `bugz-31` Refuted).
  - Why it matters: standing up an owner-scoped SQLite table is materially harder and error-prone than on Postgres — the author must hand-mirror Drizzle column types (boolean→integer mode, quoted identifiers, timestamp defaults) into a generated artifact. Dialect-parity ergonomics gap; the SQLite dialect leaks generated-artifact maintenance onto app authors.
  - Repro evidence: `notes-sqlite/src/_kovo/app-runtime-db.ts` required hand-added CREATE TABLE DDL + `SCHEMA_TABLES` entries; `notes-pg` needed only `SCHEMA_TABLES` entries. Both inspected first-hand.
  - Acceptance: the SQLite runtime-db derives `SCHEMA_DDL` from `schema.ts`/`SCHEMA_TABLES` the way the PG template does (or a build step regenerates the artifact), so adding an owner table to `schema.ts` requires no hand-written DDL and the census metadata cannot drift from the schema.
