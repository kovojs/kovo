**Meta-theme — "push enforcement to the engine" is necessary but NOT self-executing: the engine is the sole door only
if the PROVISIONING that sets it up is COMPLETE, and provisioning is itself a static enumeration that has the same
enumerate-and-allow blind spots the whole arc is about.** Round 11 confirms the engine-choke thesis holds for the
enumerated shapes (13 of 14 read shapes + all cross-owner writes were sound), then finds the pattern reproduced at TWO
new boundaries:

1. **The RLS provisioner enumerates `{owner, ownerVia-with-owner-parent}` base tables** and misses (a) VIEWS — a
   definer-semantics view (Postgres DEFAULT) granted to `kovo_reader` runs as the view OWNER (superuser) and bypasses
   RLS _and_ column-REVOKE (B1); (b) an ownerVia child whose parent is a NON-owner table gets the engine GRANT but its
   RLS policy is silently skipped — grant-without-RLS = default-allow (B2).
2. **The DEC-B1 "sole-door" lint enumerates `{static import declarations from the runtime-db module specifier}`** and
   misses (a) an aliased re-export of `appRuntimeDbProvider` through the app's own `db.ts` (B3); (b) a `dynamic
import()` of the raw driver (B4) — both open an UNCONFINED superuser connection, anonymous leak on a green build.

The through-line, restated for this round: **a choke is sound only if every object reachable by the app roles is either
a base table under FORCE RLS or a view/function PROVEN security-safe — and only if every path that can construct a DB
handle is covered by the lint.** Both are completeness obligations the current implementation approximates by
enumeration. The definer-view (B1) is the sharpest: it shows the engine's OWN default (views run as definer) is a
shape-dependent bypass, so "the engine enforces shape-independently" (followup-6 §2 C4) is false without a provisioning

- posture gate on views/definer functions.

Baseline: fresh default-Postgres (PGlite) starters linked to the local framework, built + served with
`KOVO_PARANOID=1`. Each finding independently reproduced by a skeptical verifier; all four roots re-verified first-hand
in framework source, and B1's engine mechanism re-verified first-hand on raw PGlite.

## Issues

- [ ] **B1 — A definer-semantics VIEW (Postgres DEFAULT `security_invoker=off`) over an owner table, granted to `kovo_reader`, bypasses RLS owner-scoping AND column-REVOKE: it returns every owner's rows including secret columns to any principal, on a green `KOVO_PARANOID=1` build. The framework never emits `security_invoker` and its boot posture check never inspects views/definer functions.** (HIGH, framework/security, architectural; `pg-cross-owner-allshapes`; reproduced + engine-mechanism self-verified, paranoid-confirmed)
  - Observed: default Postgres app, `orders = kovo({owner:'userId', secret:['secretNote']})`, seeded for Alice and Bob. As Alice, 13 of 14 read shapes returned only Alice/0 (sound). The exception: a plain `CREATE VIEW orders_definer_view AS SELECT id,"userId",secret FROM orders; GRANT SELECT … TO kovo_reader` returned BOTH rows including Bob's — and the secret column. The sibling `CREATE VIEW … WITH (security_invoker=true)` did NOT leak. Served on a green `KOVO_PARANOID=1 build:prod`.
  - Root cause (self-verified in source + on raw PGlite): Postgres views default to `security_invoker=OFF` — underlying table access runs as the VIEW OWNER (the DDL/seed/superuser connection, e.g. PGlite `postgres` or a `BYPASSRLS` provisioning role on managed PG), which is not filtered by `ENABLE/FORCE ROW LEVEL SECURITY` and not subject to the column REVOKE. (1) `grep -rc security_invoker packages/server/src` = 0 — the framework never emits it. (2) `checkRuntimeDbPosture` (`packages/server/src/postgres-runtime.ts:462-640`) queries only `pg_class` (`relrowsecurity/relforcerowsecurity`) and `pg_policies` — it has NO check that any reader/writer-accessible view or `SECURITY DEFINER` function over an owner table is `security_invoker`/security-safe. (3) `applyPostgresReaderColumnPrivileges` (`:1214`) iterates only Drizzle `pgTable`s, so app-added views are never auto-managed. First-hand engine repro (raw PGlite): a definer view granted to a non-superuser role returned `[{alice,ALICE-SECRET},{bob,BOB-SECRET}]` while the base table under RLS denied and the `security_invoker` sibling denied.
  - Why it matters: **cross-owner IDOR + secret-column leak through a VIEW — the single most common SQL abstraction — on a green paranoid build.** It falsifies followup-6 §2 C4 ("the engine enforces shape-independently, keyed on role/principal"): a view IS a shape, and the engine enforces it by the view-owner's role. It also CORRECTS a prior mis-refutation: `claude-bugz-30:52` and `claude-papercuts-28:27` marked "PG definer-view / SECURITY DEFINER bypass — refuted"; those refutations only held because the test view used `security_invoker=true` or was not reader-granted. The framework's OWN acceptance test (`packages/create-kovo/src/index.build.test-support.ts` addPostgresParanoidPhase5DogfoodProof) uses `security_invoker=true` — proving the requirement is known but unenforced for app-authored views.
  - Repro evidence: `/Users/mini/kovo-dogfood-round11/pg-allshapes` (definer view via `seedSql`/migrate, granted to `kovo_reader`); `KOVO_PARANOID=1 build:prod` exit 0; `curl /api/read-as-alice` → both rows + secret. Raw-PGlite mechanism probe: `scratchpad/definer-view-probe.mjs` (definer view leaks both owners + secret; invoker view + base table do not).
  - Acceptance: provisioning + boot posture must make definer objects sound — either (a) `kovo db provision` forces `security_invoker=true` on every reader/writer-accessible view over an owner/secret table and rejects `SECURITY DEFINER` functions that read them, and `checkRuntimeDbPosture` fails closed if any such object is definer-semantics; or (b) the app roles are never granted on app-authored views unless proven security-safe. A paranoid test: a reader-granted definer view over an owner table must not serve another owner's row (or must fail the boot posture check). Reclassify the engine-choke guarantee: "sole door" requires "every object reachable by the app roles is FORCE-RLS base table or proven security_invoker."

- [ ] **B2 — An `ownerVia` child whose PARENT is a non-owner table receives the full reader+writer engine GRANT but its RLS policy is silently SKIPPED (grant without `ENABLE ROW LEVEL SECURITY`): cross-owner read AND write fail-open on a green `KOVO_PARANOID=1` build.** (HIGH, framework/security, architectural; `pg-writer-grant-and-floors`; reproduced, paranoid-confirmed)
  - Observed: an `ownerVia` table pointing through a parent that is not itself `kovo({owner})` is fully readable and writable across owners — the engine applies no owner predicate because the table has no policy and RLS was never enabled on it.
  - Root cause (self-verified): `packages/server/src/postgres-runtime.ts`. The GRANT loops add every `ownerVia` child to the readable/writable sets UNCONDITIONALLY (`:1323`, `:1336` — `for (const tableName of metadata.ownerViaSourcesByTable.keys()) writableTables.add/readableTables.add`). But the POLICY loop (`:1374-1380`) does `const parentOwner = metadata.ownerSourcesByTable.get(ownerVia.parentTable); if (parentOwner === undefined) continue;` — when the parent is not an owner table, it `continue`s BEFORE the `ALTER TABLE … ENABLE/FORCE ROW LEVEL SECURITY` + `CREATE POLICY`. Net: the child table has the engine GRANT, no RLS, no policy → Postgres returns all rows (grant fully effective) → default-allow. The census/posture path does not catch it (the ownerVia entry with an unresolved parent-owner is skipped symmetrically).
  - Why it matters: the exact "grant without policy" default-allow gap the reader/writer default-deny model was meant to preclude — an owner-scoping annotation (`ownerVia`) that SILENTLY provides NO scoping when its parent chain doesn't terminate in an owner, with no build error and no boot refusal. A one-line schema shape (a plausible membership/hierarchy misconfiguration) turns owner-scoping off.
  - Repro evidence: `/Users/mini/kovo-dogfood-round11/pg-writer-grant-and-floors`; served cross-owner read+write of the ownerVia child under `KOVO_PARANOID=1`. Source: grant `postgres-runtime.ts:1323/1336` unconditional; policy `:1380` `continue` on `parentOwner === undefined`.
  - Acceptance: an `ownerVia` whose parent is not an owner (or whose owner chain cannot be resolved to a principal predicate) must be a BUILD ERROR (KV414), and provisioning must never GRANT a table it cannot policy. `checkRuntimeDbPosture` must fail closed if any granted table lacks FORCE RLS + a live policy. A paranoid test: an ownerVia-of-non-owner-parent read returns 0 / build fails, never all rows.

- [ ] **B3 — The DEC-B1 superuser-door lint is bypassed by an ALIASED RE-EXPORT of `appRuntimeDbProvider` through the app's own `db.ts`: an endpoint importing the alias calls the undefined-request path, gets the UNCONFINED superuser internal client, and leaks secrets/cross-owner rows anonymously on a green build.** (HIGH, framework/security, architectural; `pg-superuser-door-and-module`; reproduced, paranoid-confirmed)
  - Observed: `db.ts` adds `export { appRuntimeDbProvider as getRawDb } from './_kovo/app-runtime-db'`; an endpoint does `import { getRawDb } from './db'; const db = getRawDb();` (no request arg) → the internal superuser framework DB → reads any owner table / secret column with no principal, served to an anonymous caller. `KOVO_PARANOID=1 build:prod` stays green (the lint never fires).
  - Root cause (self-verified): `packages/drizzle/src/static.ts` `runtimeDbProviderImportLocalNames` (`:1464-1477`) collects local names ONLY from `sourceFile.getImportDeclarations()` whose module specifier `isRuntimeDbModuleSpecifier(...)` (i.e. the `_kovo/app-runtime-db` path). An import of `getRawDb` from `./db` does not match that specifier, so the alias is never added to the recognized-names set (`:1456`), and the call `getRawDb()` is not flagged as an `appRuntimeDbProvider()` invocation. The lint tracks direct imports of the runtime-db module, not transitive re-exports through app modules.
  - Why it matters: the DEC-B1 lint is the sole guard on the superuser internal client (followup-6 DEC-B); a single natural re-export through the app's own `db.ts` (which the template already blesses as the app DB module) defeats it → anonymous cross-owner + secret leak. The "sole door" lint is itself an incomplete enumerator — the arc's pattern, now in the guard meant to close it.
  - Repro evidence: `/Users/mini/kovo-dogfood-round11/pg-superuser-door-and-module`; green `build:prod`; anonymous `curl` returned secret/cross-owner rows via the aliased handle. Source: `static.ts:1464-1477` (specifier-scoped name resolution), `:1446-1456` (call detection keyed on those names).
  - Acceptance: the lint must resolve `appRuntimeDbProvider` through re-exports (follow `export … from` chains and local re-exports), or — stronger — the superuser internal client must not be a reachable export at all (the request-undefined path should require a framework-internal capability token, not just "call with no args"). A paranoid test: an endpoint importing an aliased/re-exported provider and calling it with no request is a build error or yields a fail-closed handle.

- [ ] **B4 — The `ENDPOINT_RAW_DRIVER_MODULES` lint scans STATIC imports only: a `dynamic import()` of `@electric-sql/pglite` (or the raw pg driver) in an endpoint opens a SECOND unconfined superuser connection to the app database, bypassing the entire role/RLS model — anonymous cross-owner + secret leak on a green build.** (HIGH, framework/security, architectural; `pg-superuser-door-and-module`; reproduced, paranoid-confirmed)
  - Observed: an endpoint does `const { PGlite } = await import('@electric-sql/pglite'); const raw = new PGlite(process.env.KOVO_DATA_DIR ?? '.kovo/pglite'); … raw.query('SELECT … FROM orders')` → a fresh superuser connection to the same data dir, reading every owner's rows + secrets with no principal. `KOVO_PARANOID=1 build:prod` stays green.
  - Root cause (self-verified): `packages/drizzle/src/static.ts:1294-1295` — the raw-driver check finds a match via `sourceFile.getImportDeclarations().find(candidate => ENDPOINT_RAW_DRIVER_MODULES.has(candidate.getModuleSpecifierValue()))`. `getImportDeclarations()` returns only STATIC `import` declarations; a `dynamic import()` is a `CallExpression` (`ImportKeyword`), never an ImportDeclaration, so it is invisible to the lint. Same blind spot as B3 (static-import-only enumeration).
  - Why it matters: the raw-driver sole-door lint (followup-6 DEC-B / the endpoint choke) is trivially bypassed by `await import(...)`, a standard dynamic-import idiom. Because the app DB is an on-disk PGlite the endpoint can re-open directly as superuser, there is no role to scope it — a total bypass of the engine choke, anonymously served.
  - Repro evidence: `/Users/mini/kovo-dogfood-round11/pg-superuser-door-and-module`; green `build:prod`; anonymous `curl` returned all owners' rows + secrets via the dynamically-imported second connection. Source: `static.ts:1294-1295` (static-imports-only), `ENDPOINT_RAW_DRIVER_MODULES` `:1249`.
  - Acceptance: the lint must also scan `import()` call expressions and `require(...)` for raw-driver specifiers; and/or the endpoint runtime must forbid opening a second connection to the app data dir (a runtime guard on the driver, not only a build lint). A paranoid test: an endpoint dynamically importing the raw driver is a build error or cannot reach the app data.

## Refuted / Not Carried Forward (encouraging — the thesis held for these)

- **13 of 14 read shapes SOUND** on Postgres under paranoid: builder `.select()`, builder `.where(bob)` (0 rows), RQBv2 `db.query.findMany/findFirst`, raw `sql`/`rawRead`, raw `.where(bob)` (0 rows), subquery-in-FROM, UNION/compound, CTE (`WITH`), JOIN from a non-owner table, column alias, ownerVia child (with owner parent), and the exported `readonlyAppDb` singleton in an endpoint — **all returned only the caller's rows / 0**. This is the round-10 `readonlyAppDb`/`db.query.*` SQLite leaks (`bugz-31` B1/B2) now FAILING CLOSED on Postgres — the engine-choke thesis working exactly as designed for base-table shapes.
- **All cross-owner WRITES denied**: update/delete of another owner's row → 0 rows; insert with someone else's owner id → `WITH CHECK` deny; ownership reassignment → deny. **Secret columns**: engine `permission denied`/KV433 on both builder and raw reads (column-REVOKE holds for the reader role).
- **OV-2 (authzPolicy string grants read+write with no RLS) — refuted**: this is the documented DEC-I honesty boundary — `authzPolicy` is app-supplied predicate responsibility, FORCE-RLS + policy-present is guaranteed, predicate correctness is the app's. (Its lack of a loud warning like the SQLite quarantine is a papercut → `papercuts-30`.)
- **Census default-deny for unclassified tables — held**: an endpoint reading an unclassified table via `kovo_reader` engine-denied (`permission denied`), confirming `bugz-31` B3 is demoted to defense-in-depth on Postgres (followup-6 DEC-B2).
