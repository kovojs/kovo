# Durable Tasks Soundness Bugz 22

Created 2026-06-30. Source of truth remains `SPEC.md` §9.6 (durable tasks & scheduling). This ledger
records security/soundness defects found while dogfooding the freshly-landed durable-task surface
(`task()` / `request.schedule()` / cron / JobRunner) as an app author. Papercuts from the same sweep
are in `plans/claude-papercuts-20.md`.

## Scope

Exercised the durable-task surface against the default postgres/PGlite `create-kovo` scaffold (linked
to the local monorepo), on the production node-preset artifact (`kovo build` → `dist/server/server.mjs`).
Tracks: schedule semantics, runner/retry/dead-letter, cron, lineage backstops, mutation atomicity,
static-analysis/audit coverage, and the SQLite fail-closed gate. Durable tasks are **fresh ground** —
no prior ledger (`papercuts*`, `papercut-super-*`, `bugz-1..21`) covers §9.6, so these are novel. The
durable-task RUNTIME mechanics are largely sound (see the encouraging refutations in the papercuts
ledger); the soundness holes below are in static audit coverage, the runaway backstops, build-time
validation, and silent enqueue.

Each finding's root cause was confirmed first-hand by reading the cited framework source; runtime
symptoms were reproduced by an authoring agent and an independent skeptical verifier (with positive
controls) in a then-healthy build env. See the env note in §Latest Verification.

## Issues

### A. Durable tasks bypass the entire static write-audit surface

- [x] **B1 — A `task()` body that writes through module-level `appDb` (the Drizzle builder) escapes KV330/KV414/KV438/KV407 and every `kovo explain` audit surface: a green build ships an un-inspectable write to a governed table.** (HIGH, framework/soundness; found by `dt-analyzer`, verified independently)
  - Observed: a task whose `run()` imports the app's `appDb` and does `appDb.insert(contacts).values(...)` (bypassing the SPEC §9.6-mandated `ctx.runMutation`) **builds green, runs green** in the prod JobRunner, and the write lands (`contacts` count 3→4) with **no KV diagnostic**. `graph.json` has no `tasks` key; `kovo explain --endpoints/--access` show no task rows; `kovo explain --unscoped` reports `total=0` despite the live direct write to a `kovo({ domain, owner })`-governed table. The same direct `appDb.insert` is also clean inside an `endpoint()` handler.
  - Root cause: `packages/compiler/src/scan/parse.ts:134-141` only collects `mutation(...)` calls into `mutationHandlers`; `task(...)` bodies are never parsed into the model. The write-audit gates (KV330 `validateDirectDbAccess` in `packages/compiler/src/validate/component-contracts.ts`, and KV414/KV438/KV407 write-provenance in `packages/drizzle/src/static/derivation.ts`) are anchored to the mutation registry, so they receive no task input. `packages/server/src/task.ts:118-122` deliberately withholds a raw `db` from `ctx` to force `runMutation`, but nothing blocks or analyzes the `appDb` module import, so the SPEC §9.6 "task DB writes MUST go through `ctx.runMutation`" MUST is unenforced and unverifiable.
  - Why it matters: a green deploy gate ships a write surface that the IDOR (KV414), mass-assignment (KV438), and invalidation (KV407) gates cannot see. An owner-scoped/secret auth table (`session`/`account`, `key:userId`, `secret:[token]`) is writable by the identical path, and a query reading the table serves stale data because KV407 never sees the write. This is the inverse of prior KV414 findings (which were too strict _inside_ mutations); here the audit does not cover non-mutation `appDb` writes at all.
  - Repro evidence: `kovo build` GREEN, no diagnostics; POST schedule `mode=directwrite` → task runs → `contacts` 3→4 + `task_proofs` row, never through the audited mutation; `kovo explain --unscoped` → `SUMMARY total=0`; `graph.json` top keys lack `tasks`.
  - Acceptance: the compiler parses `task()` bodies into the build model and applies KV330/KV414/KV438/KV407 (and KV422) to task writes; a task that writes a governed/owner/secret table without `ctx.runMutation` (or without the appropriate owner-scope/justification) fails `kovo build`, and task→table writes appear in `kovo explain`. (Name a focused compiler test asserting a task `appDb.insert(ownerTable)` fails the build.)
  - Fixed evidence: `pnpm exec vitest --run packages/compiler/src/direct-db.test.ts packages/compiler/src/registry.test.ts packages/cli/src/index.kovo-explain.test.ts` proves task `appDb.insert(...)` fails KV330 and task composition edges appear in the graph/explain surface; `pnpm exec vitest --run packages/cli/src/index.kovo-build.test.ts -t "counts source-derived component query consumers"` proves real build `graph.json` includes `tasks`.

### B. The §9.6 runaway backstops are inert in the shipped runner

- [x] **B2 — The lineage generation ceiling AND the self-reschedule delay floor are dead code in the production node-preset JobRunner: a self-rescheduling task loops forever at the poll cadence with no dead-letter and no diagnostic.** (HIGH, framework/soundness+DoS; found by `dt-lineage`, verified independently)
  - Observed: a task that self-reschedules via `ctx.schedule(self, …, { afterMs: 0 })` with `maxGenerations: 3` ran **25 times in ~2.8s** (~104ms apart = the 100ms poll interval, not a 1s floor), stopping only at the app's own safety cap — never at the framework ceiling. Server stderr showed no `exceeded maxGenerations`, no dead-letter, no `onError`.
  - Root cause: two `ctx.schedule` implementations exist and the backstopped one is dead in production. `packages/server/src/task-runtime.ts:140-150` **always** supplies `hooks.schedule`, which calls `enqueueScheduledTask` → `taskRunAt` (`task-runtime.ts:239-246`): it applies **no delay floor** and passes **no `generation`, no `lineage`**. `PostgresDurableTaskQueue.enqueue` then defaults `generation=0` and `lineage=<fresh job id>`. The backstopped default schedule hook in `packages/server/src/task-runner.ts:246-263` (which propagates `generation=job.generation+1`, `lineage=job.lineage`, applies `generationStatus()` ceiling + `selfRescheduleRunAt()` floor, default `1000`ms) runs **only when `hooks.schedule` is undefined** — so it never runs in the shipped node preset. The passing unit test `task-runner.test.ts:240` constructs the runner _without_ `hooks.schedule`, exercising the dead path.
  - Why it matters: fail-open soundness + DoS. The §9.6 backstops that bound infinite lineages and DB hot-loops are completely inert. The default `maxGenerations:64` gives zero protection (every reschedule resets generation to 0 and starts a fresh lineage), so any self-rescheduling task — saga step, poll-until-done, or a buggy unconditional reschedule — runs forever hammering `_kovo_jobs`. Lineage-based dead-letter observability is also broken (every child gets a new lineage id). Green-build/dead-artifact skew: the framework has the correct impl + a passing test, but production is wired to bypass it (cf. `bugz-17` B2 regression class).
  - Repro evidence: `tsc --noEmit` exit 0, `build:prod` GREEN; POST `mode=self-loop` → count climbs 5/10/15/20/25 at ~100ms cadence, flat at 25 (app cap); no dead-letter/error line in stderr.
  - Acceptance: the production schedule path (`task-runtime.ts` `hooks.schedule`) propagates `generation`/`lineage` and applies the delay floor + generation ceiling, so a `maxGenerations:N` self-reschedule dead-letters at the ceiling with a diagnostic and `afterMs<floor` is clamped. A test must exercise the _production_ `hooks.schedule` wiring, not the default fallback.
  - Fixed evidence: `pnpm exec vitest --run packages/server/src/task-runtime.test.ts packages/server/src/task-runner.test.ts` proves the node runtime uses the runner-backed `ctx.schedule` path with lineage/generation and the runner clamps self-reschedule delay/dead-letters at `maxGenerations`.

### C. Build-time validation gap → permanent outage

- [x] **B3 — An invalid cron string (6-field, or `@daily`) passes `task()` validation AND a green `kovo build`, then permanently bricks the ENTIRE app — every route 500s — via an unwrapped first materialize + a cached rejected start promise.** (HIGH, framework/soundness+availability; found by `dt-cron`, verified independently with a control)
  - Observed: setting `cron: '0 0 * * * *'` (6-field) or `'@daily'` → `build:prod` GREEN, but the booted prod artifact returns HTTP **500 for every request** (`/api/health`, `/`, `/login`, `/api/*`) and never recovers. Control: a valid `'* * * * *'` rebuild → `/api/health` 200, `/` 303, `/login` 200, proving the invalid cron is the sole cause.
  - Root cause: `packages/server/src/task.ts:186-206` `assertTaskCronOptions` validates only that `cron` is a non-empty _string_ — not the five-field grammar its own JSDoc documents (`task.ts:84`); `kovo build` does no cron validation. `packages/server/src/task-cron.ts:200-214` `parseCronExpression` throws lazily at runtime when `fields.length !== 5`. `packages/server/src/task-runtime.ts:91` — the first `await this.materializeRecurringTasks()` in `start()` is **not** wrapped in try/catch (unlike the periodic tick at `:163-169`, which swallows), so `start()` rejects; `ensureStarted` (`:73-77`) caches the rejected promise via `??=` and never sets `this.runner`, so every later request re-awaits the rejection. `app.ts:332-335` awaits `ensureStarted` before dispatch for **every** request → all routes 500 (handled gracefully as a 500 doc, not an unhandled crash). See the shared amplifier in `claude-papercuts-20.md` §E1.
  - Why it matters: a config typo using the two most common cron dialects clears the deploy gate and then takes down the whole app (health probe, login, every mutation), permanently, requiring a redeploy. Green-build-dead-artifact with wildly disproportionate, unrecoverable blast radius; SPEC §9.6:1034 mandates build-time fail-closed with an actionable diagnostic.
  - Repro evidence: invalid 6-field cron → `build:prod` GREEN + emitted `handler.mjs` contains `cron: "0 0 * * * *"`; booted artifact → `/api/health`, `/`, `/login`, `/api/task-proof-count` all 500; valid-cron control all 2xx/3xx.
  - Acceptance: `task({ cron })` (and `kovo build`) validate the five-field grammar and reject `@daily`/6-field with an actionable build-time diagnostic; OR a materialization failure is contained to the offending cron task and never bricks unrelated routes. Mitigant: requires author error, not externally triggerable.
  - Fixed evidence: `pnpm exec vitest --run packages/server/src/task.test.ts packages/server/src/task-cron.test.ts packages/server/src/task-runtime.test.ts` proves `task({ cron })` rejects 6-field/`@daily` cron before build output and task-runtime startup failures no longer permanently brick unrelated routes.

### D. Silent durable-effect data loss

- [x] **B4 — `request.schedule()` of a task not registered in `createApp({ tasks })` commits a durable `_kovo_jobs` row that the claimer never selects — stuck `ready` forever, no error, no dead-letter.** (MED, framework/soundness; found by `dt-analyzer`, verified independently with a control)
  - Observed: scheduling a defined-but-unregistered task builds GREEN; the schedule **commits** (HTTP 303, `_kovo_jobs` row `status='ready'`, a `TaskHandle` is returned), but the job never runs, never errors, never dead-letters — proof count stays 0 and the row stays `ready` for the whole poll window. A registered control task drains in ~3s, proving the runner is live.
  - Root cause: `packages/server/src/task-runner.ts:102-132` — `claimAvailable` filters `claimDue({ taskKeys })` to `claimableTaskKeys`, built **only** from the registered task set, so an unregistered `task_key` is never claimed. The unknown-task dead-letter branch (`task-runner.ts:163-173`) only runs for an _already-claimed_ job, so it is unreachable for a same-deploy unregistered job. `task-runtime.ts:60-69`/`enqueueScheduledTask` enqueue by `definition.key` with no registration check, and `build.ts` only errors on tasks-without-JobRunner (KV445/KV446), not on `schedule(X)` where `X ∉ tasks`.
  - Why it matters: a durable scheduled side-effect the author believes is enqueued (green build, 303, handle returned) silently never executes — no build error, no runtime error, no dead-letter — for any forgotten-registration deploy. Contradicts SPEC.md:1016 ("commit means the job is ready to run") and :1035 ("must never silently enqueue work that no deployed artifact can run"). Distinct from KV445 (missing JobRunner, which fails closed at build).
  - Repro evidence: `mode=unregistered` → 303; poll 8s → count 0 every tick; jobs-inspect shows `{ unregistered, ready, 1 }` stuck; registered control drains to 1 in 3s.
  - Acceptance: `request.schedule(def)`/`ctx.schedule(def)` of a task not in `createApp({ tasks })` fails closed — a `kovo build` error (the `tasks:[…]` literal and `def.key` are statically known) or a throw at schedule time — never a silently-undrainable committed job.
  - Fixed evidence: `pnpm exec vitest --run packages/server/src/task-runtime.test.ts packages/server/src/task-runner.test.ts` proves both `request.schedule` and `ctx.schedule` throw `UnknownDurableTaskError` before enqueue for tasks outside `createApp({ tasks })`.
