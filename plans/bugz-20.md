# Bugz 20

Created 2026-06-29. Source of truth remains `SPEC.md`. Three confirmed security/soundness defects
escalated from the postgres-template dogfood sweep (companion papercuts ledger:
`plans/papercuts-super-9.md`), reproduced against a clean `create-kovo` (default postgres) build on
`main`. B1 (raw-SQL write IDOR bypass) was self-verified first-hand from source.

- **B1 — fail-OPEN security:** a raw-SQL write via the documented `db.execute(sql\`…\`)` escape is
  invisible to write extraction, so the KV414 (IDOR) and KV438 (mass-assignment) blocking gates
  silently pass — directly violating the normative SPEC §10.3:1167 raw-SQL-write contract.
- **B2 — data-integrity soundness:** the no-JS idempotency replay omits the body fingerprint, so a
  same-token re-submit with a different body silently replays the prior commit and drops the new
  write (silent lost-update), violating SPEC §9.1:1184.
- **B3 — secret-to-wire:** the scaffold ships Better Auth credential tables unclassified, so an
  owner-scoped query projecting `account.password` builds green and ships the scrypt hash to the
  client.

## Scope

- Apps: fresh `create-kovo` default (postgres/PGlite) scaffolds, link-local to the monorepo,
  exercised through `vp dev`, `build:prod`, and authenticated HTTP drives.
- Out of scope: the idempotency-status / SQL-ergonomics / build-hermeticity / routing papercuts (in
  `papercuts-super-9.md`).

## Issues

- [x] **B1 — A raw-SQL write via the documented `db.execute(sql\`UPDATE…\`)` escape is invisible to write extraction: it produces no write/touch/owner facts, so KV406 never forces `touches`/`tables` and KV414 (IDOR) + KV438 (mass-assignment) silently pass — fail-open versus the typed builder, which is correctly gated.** (HIGH, framework; found by `t3-sql-surface`)
  - Exploit / observed: a registered mutation doing `request.db.execute(sql\`UPDATE contacts SET company = ${company} WHERE id = ${id}\`)` (with `id` a client-visible input) builds **green** — no KV406/KV402/KV310; `dist/.kovo/graph.json` shows the mutation `writes=None, invalidates=None`. The byte-equivalent typed write `db.update(contacts).set({company}).where(eq(contacts.id, id))` flips the build to fatal (KV402 "write touched an undeclared domain" + KV310). With `owner:(t)=>t.userId` on the table and `touches:[contact]` declared on the raw write, the **typed** write correctly fires `KV414 WRITE … (IDOR)` for the arg-keyed owner row, while the **raw `db.execute`** write fires only KV310 and **zero KV414** — an owner-table write keyed by client-visible input reaches the row with no proven session predicate. So an author who renames/updates/deletes an owner-scoped row through raw SQL gets no IDOR audit and no mass-assignment check.
  - Root cause: `packages/drizzle/src/static.ts:138-141` lists `execute` in `UNCLASSIFIED_DRIZZLE_RECEIVER_MUTATION_METHODS` (`{$count, execute}`), and `CLASSIFIED_DRIZZLE_RECEIVER_METHODS` (`:156-166`) excludes it (only `select*/delete/insert/refreshMaterializedView/transaction/update/with`). So `db.execute(sql\`…\`)` is never classified as a write site: the unclassified-call extractor (`project-receivers.ts:1070-1130`) yields no KV406 write fact, no touch/predicate/value facts, hence nothing for `scopeAuditsFromQueryFacts`/KV414 or KV438 to audit. This directly violates SPEC §10.3:1167, which is normative: a raw-SQL write "MUST additionally declare `tables:`… which the runtime executor parses and enforces… on a production write to a table outside the declared `tables:`, the executor MUST fail closed" — i.e. raw writes are supposed to be KV406-blocked and gate-audited, not silently unclassified.
  - Why it matters: KV414/KV438 are the framework's marquee write-side security gates (the "secure by construction" headline), and the raw-SQL escape is documented as a first-class write form. The gates are bypassable by routing a write through `db.execute(sql\`…\`)` — and authors are actively pushed toward that path because Kovo's own safe `sql`/`staticSql` constructors are unusable in `db.execute` (see `papercuts-super-9` B1 / `sql-1`). A cross-tenant UPDATE/DELETE keyed by a client-supplied id ships through a green `kovo build` with no IDOR diagnostic.
  - Repro evidence (self-verified, source): `static.ts:138-141` (`execute` UNCLASSIFIED) + `:156-166` (`execute` absent from CLASSIFIED); behavioral repro in `t3-sql-surface` — raw `db.execute` write → `graph.json writes=None`, no KV414; typed write → KV402/KV310/KV414 fire.
  - Fix plan (phased — reuses existing machinery; full static SQL analysis is NOT required and NOT promised by SPEC, whose raw-SQL model is "declare conservatively + executor fail-closed"):
    - [x] **Phase 1 — close the fail-open (security-critical, smaller half).** Move `db.execute` out of `UNCLASSIFIED_DRIZZLE_RECEIVER_MUTATION_METHODS` (`static.ts:138-141`) and route it through the existing opaque-write → KV406 path (the "degrade to KV406 instead of disappearing" idiom already pervasive in `domain-writes.ts`/`project-receivers.ts`/`receiver-surface.ts`), so a raw write can no longer build with `writes=None`: it is forced to declare `touches` + `tables` (SPEC §10.3:1167). This also restores the KV407 invalidation edges. Handle `execute`-as-read (`db.execute(sql\`SELECT…\`)`) and DDL/seed (`client.exec`) conservatively via the declare-or-fail-closed model — do NOT treat `execute` as unconditionally a write. Acceptance: a raw `db.execute(sql\`UPDATE owner_table…\`)` with no `touches` fails KV406; a read-only `execute` and the starter DDL/seed still build.
      - Evidence: `pnpm exec vitest --run packages/create-kovo/src/index.build.test.ts -t "raw .*db.execute writes"` covers scaffold KV406 for undeclared mutation raw writes.
    - [x] **Phase 2 — make KV414 audit the raw owner-table write (moderate; one design decision).** A raw `WHERE` is not statically provable as session-scoped, so adopt the Kovo honesty-boundary pattern: a raw write whose declared `tables:` includes an `owner`-annotated table **fails KV414 closed UNLESS** it is wrapped in the existing `trustedSql(statement, { justification })` attestation (`runtime.ts:153-160`) — routing the unsafe construction through the already-built justifying constructor with a source-visible reason a reviewer/gate can see. Acceptance: a raw owner-table write keyed by an arg fires KV414 with no attestation, and passes only with an explicit `trustedSql` justification — matching the fail-closed posture the typed builder enforces.
      - Evidence: `pnpm exec vitest --run packages/create-kovo/src/index.build.test.ts -t "raw .*db.execute writes"` covers scaffold KV414 for declared owner-table raw writes and trusted raw-owner build success.
    - [x] **Phase 3 (remaining runtime proof).** Add a deployed/runtime cross-tenant raw-write check, and decide whether raw `tables:` should also synthesize a KV438-style mass-assignment fact for governed columns or remain a trustedSql-only escape.
      - Decision: raw `tables:` now synthesizes fail-closed KV438 (`via=raw-sql`, `provenance=unknown`) for governed tables unless the statement is wrapped in `trustedSql(...)`.
      - Evidence: `pnpm exec vitest run packages/drizzle/src/raw-sql-static.test.ts packages/test/src/verifier-ownership.test.ts packages/test/src/pglite-harness.test.ts` proves raw KV438 extraction, trusted suppression, runtime owner-write KV414, and a real PGlite raw cross-tenant owner-write check.
      - Evidence: `pnpm exec vitest --run packages/create-kovo/src/index.build.test.ts -t "raw .*db.execute writes"` proves the production scaffold artifact reports KV406, KV414, and KV438 for raw `db.execute` writes and still accepts trusted raw SQL.
    - Pairs with `papercuts-super-9` B1 (`sql-1`): fixing Kovo's `sql`/`staticSql` constructors to type-check in `db.execute` is the prerequisite for steering authors onto a gated raw-SQL path instead of the ungated drizzle-native one. Sequence sql-1 alongside Phase 2.

- [x] **B2 — The no-JS mutation idempotency replay omits the request body fingerprint, so reusing a `Kovo-Idem` token with a different body silently replays the prior 303 and drops the new write (silent lost-update).** (MED, framework; found by `t1-concurrency`)
  - Exploit / observed: signed-in, no-JS POST-redirect-GET path against the shipped `/_m/mutations/add-contact`. Submit #1: `Kovo-Idem=X`, body `{name:'Silent A', email:silentA@…}` → 303, row inserted. Submit #2: **same** `Kovo-Idem=X`, **different** body `{name:'Silent B', email:silentB@…}` → 303→200 with a normal success page, **no error banner**, and **silentB is not inserted**. The user sees success while their distinct second submission is silently discarded. The enhanced (`Kovo-Fragment`) path correctly rejects the same idem-reuse-with-different-body as a 409 `IDEMPOTENCY_CONFLICT`.
  - Root cause: `NoJsMutationReplayStore` has no fingerprint parameter (`packages/server/src/mutation-wire.ts:245-251` `get(scope,idem)/reserve(scope,idem)`); the adapter `noJsReplayStoreFromMutationStore` (`mutation.ts:757-778`) drops the third fingerprint arg; the no-JS branch (`mutation.ts:961-971`) never computes a request fingerprint. In `createMemoryMutationReplayStore`, `fingerprintsMatch` (`replay.ts:529-530`) returns true when either side is undefined, so the undefined no-JS fingerprint makes `get`/`reserve` (`replay.ts:126,141`) never raise `MutationReplayConflictError` → any later same-idem submit replays the first commit regardless of body. The enhanced path sets `requestFingerprint = canonicalRequestFingerprint(rawInput)` (`mutation-wire.ts:333`), so it is body-sensitive.
  - Why it matters: SPEC §9.1:1184 is normative — "Token collision within (principal, source-derived mutation identity) is a server-detectable integrity fault answered as a 422 schema-class failure (§9.2), never a silent replay of an unrelated commit," and "a re-submit that edits visible fields therefore produces a distinct token — eliminating the silent lost-update." The no-JS POST-redirect-GET path is a first-class SPEC §6.3 surface; a realistic trigger is a bfcache/back-forward restore (SPEC §9.1:807) of a form with its stale hidden `Kovo-Idem`, then the user editing fields and resubmitting → their new contact/order/transfer is silently lost while the UI shows success. (Distinct from the fixed `bugz-3` L3 / `bugz-4` L5 / `bugz.md` M1 — this regression rides on top of the no-JS-dedup fix.)
  - Repro evidence: two no-JS POSTs, same `Kovo-Idem`, different bodies → second 303, handler ran once, only the first row inserted; the verifier reproduced it with a focused vitest against the real `noJsReplayStoreFromMutationStore` adapter (second submit returns 303, `inserted == ['silentA']`, silentB lost; enhanced store with the fingerprint correctly throws `MutationReplayConflictError`). Source: `mutation.ts:760-777,964-971`, `replay.ts:126,141,530`.
  - Fix / acceptance: thread a body fingerprint through the no-JS replay store (parity with the enhanced path), so a same-token different-body re-submit is a server-detected conflict (per SPEC §9.1:1184, a 422 — see `papercuts-super-9` A1 for the 409-vs-422 status nuance), never a silent replay. Prove with a no-JS test: same idem + different body → conflict, both writes accounted for.
  - Evidence: `pnpm exec vitest run packages/server/src/mutation-no-js.test.ts packages/server/src/replay.test.ts packages/server/src/mutation-stale-version.test.ts packages/server/src/mutation-wire.test.ts`; prod-artifact gate `rejects no-JS add-contact idempotency token collisions`.

- [x] **B3 — `create-kovo` scaffolds the Better Auth credential tables (password hash, session/OAuth tokens) with no secret classification, so KV435 does not protect them — an owner-scoped query projecting `account.password` builds green and ships the scrypt hash to the client wire.** (MED, template; found by `t4-config-secrets`; residual of `bugz-3` M6 / `bugz-4` M7)
  - Exploit / observed: in the default starter, annotating the hand-authored `account` table with a domain+owner (the natural move to make it readable) and writing an owner-scoped query projecting `password: account.password` + `accessToken: account.accessToken` builds **green** (KV435 does not fire). After a real Better Auth login, `GET /_q/…/auth-secrets-query` returns `200` with `{"items":[{"password":"3f411b…:631362…"}]}` — the demo user's scrypt password hash shipped to the client. (Reading the tables unannotated is blocked by KV407; reading them `exempt:true` is blocked by KV411 — so the annotated path is the one an author would take.)
  - Root cause: `packages/create-kovo/templates/src/schema.ts:24-66` declares `session.token`, `account.accessToken/refreshToken/idToken/password` as plain `text()` with no `kovo({ secret })` (explicit comment at line 24 says they carry no annotation). `packages/better-auth/src/index.ts:1-32` re-exports only `type`s — `betterAuthSchemaBridge`/`annotateBetterAuthSchemaSource` (`internal.ts:8,349`) are not public, so the scaffold cannot apply the secret classification the framework uses internally. (The verifier refined the author's secondary claim: classifying the table is the *mechanism* that would make KV435 fire, but the scaffold never engages it; the runtime leak itself is confirmed.) This is the create-kovo-template residual of the `bugz-3` M6 / `bugz-4` M7 credential-to-wire class, which fixed the internal bridge but never reached the hand-authored starter schema.
  - Why it matters: the default starter ships the auth credential columns with no secret protection, so the framework's KV435 secret-to-wire backstop is silently disengaged for exactly the highest-value columns (password hash, session/OAuth tokens). An author who reads the account table (a normal thing to build a profile/admin view) leaks credentials with a green build.
  - Repro evidence: `t4-config-secrets` — owner-scoped query selecting `account.password` → `build:prod` exit 0 (no KV435); login + `GET /_q/…` returns the scrypt hash. Source: `templates/src/schema.ts:24-66`, `better-auth/src/index.ts:1-32`.
  - Fix / acceptance: the scaffold (or a public `create-kovo`-usable bridge) must classify the Better Auth credential columns as `secret`, so a query projecting `account.password`/tokens fails KV435 at build. Prove with a create-kovo build test: a query projecting a credential column fails the build.
  - Evidence: `pnpm exec vitest --run packages/create-kovo/src/index.test.ts -t "classifies Better Auth credential columns as secret"` and prod-artifact gate `blocks starter Better Auth credential projections`.
