# Compiler & Framework Hardening — Hack Audit

> Audit: 2026-06-14. Source of truth: SPEC.md. Method: 8-slice multi-agent audit (compiler ×4, drizzle, runtime, server, core/ui), every finding adversarially verified against SPEC §5.2 rule 8 carve-outs and v2/v3 deferrals. **46 findings raised → 36 confirmed (10 refuted as false positives — sanctioned parser boundary, permitted carve-outs, v2/v3 deferrals) → 32 distinct after deduping near-identical cross-slice reports.** The four CRITICAL findings below were re-verified by hand against the cited source.

## Summary

The audit confirms a consistent failure mode: **capabilities that look complete because they have wired call sites and green tests, but whose principled core is absent, narrowed to a fixture shape, or never reachable in production.** Across slices the recurring move is to substitute a name/string/duplicate-literal heuristic for a typed-fact or dataflow proof (FW201, FW301, FW320, FW221, Drizzle db-type identity, Drizzle key derivation), to ship a real algorithm as exported-but-unconstructed dead code (DOM morph, template-stamp reconciliation, route-ambiguity FW228, §4.6 merge engine), or to assert a normative invariant with a check that cannot fail (render-equivalence self-comparison). The §4.6 attribute-merge mechanism and the morph path each surfaced from multiple slices independently, confirming they are systemic gaps rather than isolated bugs.

**Cross-cutting themes:**

- **Heuristic-for-proof substitution.** A whole class of gates decides a _typed/dataflow_ property by matching _names or strings_: FW201 (8-name denylist vs. exhaustive channel membership), FW301 (state-key name prefix vs. initializer dataflow), FW320 (payload key leaf-name vs. value provenance), FW221 (module-flat id set vs. per-component scope), and the Drizzle db-receiver type _name_ vs. drizzle-orm _provenance_. Each over-fires on coincidence and under-fires on the real case.
- **Real algorithm shipped as unreachable dead code.** The genuine implementation exists and passes isolated tests, but no production seam constructs it: the idiomorph-class DOM morph (`DomMorphRoot`/`keyedDomMorph`), keyed template-stamp reconciliation (`reconcileTemplateStamp` only on test fakes), route-ambiguity detection (`findRouteAmbiguities` has zero callers), and the §4.6 merge engine (lives only in the gallery test oracle). Tests stay green against the fake/isolated path while every shipped app gets innerHTML, no list reconcile, no ambiguity error, and no merge.
- **Fixture-shaped narrowing.** Lowering/validation handles exactly the demo-corpus shape and silently drops everything else: single-attribute inline derives, non-self-closing `<Link>`, static-string `viewTransitionName`, top-level bare `eq()` key derivation, and the §4.6 merge presented as "two literal duplicate attributes on one element."
- **Verification gates that cannot fail.** The render-equivalence "semantic gate" compares a lowered string to itself, and the morph/fragment-apply suite asserts the same innerHTML behavior production uses — so the normative property each claims to guard is never actually exercised against the code path that ships.

## Findings

### Architectural shortcuts

- [ ] **[HIGH] Render-equivalence semantic gate is tautological (compares lowered source to itself)** — `packages/compiler/src/emit/server.ts:25-61,215-224`, `packages/compiler/src/compile.ts:129-130,165-167,216-222` (SPEC §5.2 rule 3)
  - **What:** `renderEquivalenceCheck(server, serverRenderedSource, executableSource)` sets `expected = serverRenderedSource` and `actual = run(renderSource())`, where `renderSource()` just wraps that same lowered string in a template literal. Both sides derive from one lowered string; `ok` is `serverRenderedSource === serverRenderedSource`.
  - **Why it's a hack:** SPEC rule 3 requires an independent differential — render the AUTHORED model and the LOWERED model, assert byte-identical HTML — so the fixpoint proves behavior preservation. This can only fail if `templateLiteral()` escaping breaks or `runInNewContext` throws; no lowering that changes emitted HTML can make `ok=false`. The `server.ts:216` comment asserts this round-trip _is_ the `render(src)≡render(compile(src))` invariant, and `fw check` reports it as enforced.
  - **Fix:** Derive the two HTML strings independently — produce the LOWERED-side HTML by executing `renderSource()` as today, and the AUTHORED-side HTML from a separate reference render over the originally parsed model (available at `compile.ts:55,63` before lowering), then diff byte-for-byte allowing only provably HTML-preserving stamp/handler-attribute deltas (`fw-c`, `fw-deps`, `fw-state`, versioned handler values). Until an authored renderer exists, stop labeling it the rule-3 gate and do not let `fw check` report the invariant as enforced.

- [ ] **[MEDIUM] FW301 decides "server fact in local state" by state-key name prefix, not dataflow** — `packages/compiler/src/validate/component-contracts.ts:36-61,282-288` (SPEC §3.5/§11.3 FW301)
  - **What:** `validateServerFactsInLocalState` fires only when a state key `startsWith(queryName)` at a word boundary (`stateKeyHasQueryPrefix`). `state: () => ({ accountNameDraft: '' })` with `queries: { account }` is flagged FW301 purely because the key spelling begins with `account`; the initializer is never inspected.
  - **Why it's a hack:** Whether local state stores a server fact is a dataflow property (does an initializer copy query data?). The name proxy over-fires on benign client fields spelled like a query (`accountNameDraft`, `cartView`) and under-fires when a server value lands in a differently-named key (`const c = cart.count; return { saved: c }` is never caught). The rule detects a naming coincidence, not the bug.
  - **Fix:** Surface each state-return entry's initializer references as typed facts in the parser (reuse the existing `PropertyAccessPathModel`/references extraction — do NOT re-parse `value` in validate, which would violate rule 8). Flag FW301 only when the initializer reads a path rooted in a declared query name, regardless of key spelling. Update `state-events.test.ts` so the `accountNameDraft` case stops firing and a `{ saved: c }`-via-alias case starts firing.

- [ ] **[MEDIUM] View-transition style merge emits a duplicate `style` attribute for `style={expr}`** — `packages/compiler/src/lower/view-transitions.ts:42-67,78-92` (SPEC §5.2 rule 3; §4.2 view-transition stamping)
  - **What:** `viewTransitionStylePatches` only matches a `style` attribute with `value !== undefined` (string literal). A `style={expr}` has no `.value`, falls through to `openingTagStyleInsertion`, and a second `style="view-transition-name: x"` is appended — `<img style={q.product.css} … style="…">` ships two `style` attributes.
  - **Why it's a hack:** Duplicate `style` attributes are invalid HTML; the browser keeps the first, so the view-transition-name is silently lost. The merge string-patches the static `.value` channel only and never models the dynamic-expression case, breaking the cross-document transition with no diagnostic.
  - **Fix:** Detect the `style` attribute regardless of static-vs-dynamic (match `name === 'style'`, then branch on `value` vs `expression`). For the expression form, merge view-transition-name into the single style channel (rewrite to `style={`${expr}; view-transition-name: x`}` or lower through a concatenating derive) so exactly one `style` is emitted. Add a regression test for `viewTransitionName` + `style={expr}`.

- [ ] **[MEDIUM] Fragment-target props type drops non-primitive props, emitting `{}` that accepts any patch** — `packages/compiler/src/graph.ts:104-114`, `packages/compiler/src/scan/parse.ts:1165-1171` (SPEC §6.2 Fragment targets; §6.1 FragmentTargets registry)
  - **What:** `fragmentTargetPropsType` keeps only props whose `staticConstructorType` is set, and `staticConstructorTypeEntry` recognizes only `String`/`Number`/`Boolean`. Any other constructor or schema-typed prop is `flatMap`ped away; if all props drop, it returns the literal `'{}'`, which the emitted `FragmentTargets['<name>']` interface treats as a near-top type accepting arbitrary props.
  - **Why it's a hack:** SPEC §6.2 requires the type system to prove a fragment patch carries the right component's props. Silently dropping unrecognized props lets a patch with missing/wrong props typecheck — a hole directly under the patch-typing proof surface, shaped to the primitive props the demo corpus happens to use.
  - **Fix:** For any prop without a recognized `staticConstructorType`, emit its declared type (reuse the captured `entry.value` source slice or emit `unknown`) instead of dropping it; ideally extend `staticConstructorTypeEntry` to resolve schema/object props. Never collapse to `'{}'` when ≥1 prop key was declared — emit a non-empty object type with `unknown` members (preserving patch completeness) and surface a diagnostic for the unresolved prop.

- [ ] **[LOW] Inline attribute-derive lowering skips any element with >1 derivable attribute, leaking raw JSX** — `packages/compiler/src/lower/inline-derives.ts:62-63` (SPEC §4.8 #2 named derives)
  - **What:** `const candidate = candidates[0]; if (!candidate || candidates.length !== 1) continue;` — when an element has two bind-able attribute expressions, BOTH are dropped. `<button aria-expanded={state.open ? …} aria-busy={state.busy ? …}>` emits literal `{…}` JSX in attribute positions with no `data-bind:*`.
  - **Why it's a hack:** §4.8 lowers every inline expression in a bound position to a named derive; nothing restricts this to one attribute per element. The `length !== 1` guard is an ad-hoc cap tuned to the single-attribute fixture. (The §4.9 pass does emit FW311 for the dropped positions, so this is reduced auto-lowering coverage, not silent staleness.)
  - **Fix:** Replace the single-candidate guard with `for (const candidate of candidates)`. Each candidate already produces an independent derive export + per-attribute `SourceReplacement` keyed by `attribute.name` with a distinct `baseName`, so emitting all of them is mechanical and clears the spurious FW311 warnings.

- [ ] **[LOW] FW301 diagnostic span drifted across the state-binding lowering refactor (reports the wrong line)** — `packages/compiler/src/validate/component-contracts.ts:51-60` (SPEC §5.2 rule 5; §11.3 FW301)
  - **What:** The two `state-events.test.ts` FW301 tests currently fail only on `start.line` (expected 4, received 6). FW301 is emitted with `diagnosticFor(…, stateObject.start, …)` against the lowered source, but state-binding lowering shifts text ahead of the state-return object and the FW301 path was never wired to the offset remap that FW311 already uses.
  - **Why it's a hack:** Fragile offset handling left un-remapped across the refactor; the diagnostic frame points into lowered IR coordinates instead of the authored TSX, violating the precise-source-frame mandate. (The lint decision still fires correctly — only the position drifted.)
  - **Fix:** Thread `diagnosticSource` and `sourceOffsetMap` into `validateServerFactsInLocalState` and emit FW301 via `generatedOffsetToOriginal(sourceOffsetMap, stateObject.start)` against the original source — mirroring `fw311Diagnostic`. Re-run `state-events.test.ts` to confirm both tests pass.

- [ ] **[LOW] Validated mutation input dropped before post-commit rerun; reconstructed from `change.input ?? rawInput`** — `packages/server/src/mutation.ts:746-748,452,781-813`, `MutationSuccess` shape `mutation.ts:65-72` (SPEC §10.3 post-commit rerun / read-your-writes)
  - **What:** `MutationSuccess` carries value/changes/rerunQueries but not the parsed input. `renderMutationResponse` rebuilds the render input as `changes.find(c => c.input !== undefined)?.input ?? rawInput`. Reruns are _scheduled_ with the validated/coerced input but _rendered_ with this reconstruction, so when a mutation invalidates only via manual `context.invalidate()` without echoing input (falling back to unvalidated `rawInput`) and the query's instanceKey derives from a coerced field, the render-time key can diverge from the scheduled `target.instanceKey` and the query silently never renders.
  - **Why it's a hack:** The lifecycle must rerun in the same request context with the validated input, but that input is structurally unavailable at render time, so the code recovers an approximation. The common registry-touch case (where `change.input` equals the parsed input) masks the gap rather than closing it.
  - **Fix:** Thread the validated coerced `input` onto `MutationSuccess` (set from the parsed input in `runMutation`) and use it directly as the rerun/render input in `renderMutationResponse`/`renderQueryChunks`/`queryMatchesRerun`, so render-time and scheduled instanceKeys derive from the same coerced input. Keep `change.input` for change-record metadata only.

- [ ] **[LOW] FW232 cannot fire on real composition — only on duplicate source attributes** — `packages/compiler/src/validate/markup.ts:205,238-240` (SPEC §4.6 aria/role/data-state author-override lint)
  - **What:** FW232 is emitted only inside the `count >= 2` loop for `aria-*` or the hardcoded `['role','data-state']` set. In the normative attrs-function flow the primitive supplies an aria/data-state attr once and the author supplies it once across two merged sources — never a duplicate on one author element — so FW232 never triggers for a genuine override.
  - **Why it's a hack:** §4.6 FW232 is the "escape hatch stays visible" lint for an author overriding a primitive-owned attribute. As duplicate-name detection it fires on a shape real composition never emits and stays silent on the exact override it should surface; `['role','data-state']` is also a hardcoded partial. (No composition output is currently wrong because the merge engine is unbuilt — see the §4.6 merge finding — so this is a forward-stub, not a shipped bug.)
  - **Fix:** When the §4.6 attribute-record merge lands, move FW232 into the merge resolver and fire it when the author value differs from the primitive-owned aria/role/data-state value on a merged element, deriving the owned set from the primitive's emitted attribute record. Until then, keep the duplicate-name path but add a TODO citing §4.6 and the Phase-2 plan item so the partial coverage is legible.

### String/regex where typed facts are mandated

- [ ] **[HIGH] Drizzle receiver "type identity" is type-NAME identity — a fake class named `PgDatabase` is accepted** — `packages/drizzle/src/static.ts:2032-2069`, `packages/drizzle/src/drizzle-surface.ts:51-53` (SPEC §11.1 step 1)
  - **What:** `isDrizzleDatabaseType` returns `drizzleDatabaseTypeNames(type).some(isDrizzleDatabaseTypeName)`, which only checks the bare symbol/alias/apparent NAME (and a `getText()` regex fallback) against `{NodePgDatabase, PgDatabase, PgliteDatabase, PostgresJsDatabase}` ∪ `/^Neon.*Database$/`. The declaring module is never verified. Empirically a `class PgDatabase` in `./fake` with no drizzle import was accepted and produced a real `{domain:'cart'}` touch.
  - **Why it's a hack:** §11.1 step 1 grounds the pass on type identity that ORIGINATES IN drizzle-orm precisely so renames/structural fakes are irrelevant. Name matching means any project type named `PgDatabase`/`Neon*Database` is treated as a Drizzle db (its writes extracted as authoritative invalidations), and a re-exported/renamed real db type that prints differently is missed. The table-factory path already verifies provenance (`isDrizzleOrmDeclaration`), so the db-receiver path is an unprincipled shortcut.
  - **Fix:** Resolve the receiver type's symbol/apparent/alias/base symbols to declarations and require at least one declaration's file path to include `drizzle-orm` (or its import specifier to start with `drizzle-orm`) — mirror `isDrizzleOrmDeclaration` exactly as the table-factory path does. Only after that provenance gate passes should the name disambiguate the db flavor. Drop `drizzleDatabaseTypeNameFromExactTypeText`; when the symbol is unresolvable, treat the receiver as unresolved and emit FW406 (manual touches) per §11.1 step 2E.

- [ ] **[MEDIUM] CSS scope-fallback selector prefixing is a fragile comma-splitter that corrupts `:is()`/attribute commas and skips nested rules** — `packages/compiler/src/css.ts:145-180` (SPEC §13.1 tag-prefixed donut-scoped fallback)
  - **What:** `prefixCssBlockSelectors` splits each rule's selector on raw commas (`selector.split(',')`) and prefixes per part. `:is(.a, .b) span` becomes `[fw-c="x"] :is(.a:not(…), [fw-c="x"] .b) span…` (comma inside `:is()` split, prefix injected inside the pseudo); `[data-x="a,b"]` mangles the attribute value; `.card { & .title { … } }` emits the nested body verbatim with no host prefix or donut exclusion.
  - **Why it's a hack:** §13.1 requires the fallback to be valid, donut-scoped CSS for engines without `@scope`. Bare-comma splitting misreads commas inside `:is()/:where()/:not()/:nth-child(... of ...)` and attribute strings, and leaves nested-rule bodies unscoped. The `@scope` path is fine because it wraps the body verbatim; only the legacy fallback is broken.
  - **Fix:** Replace `selector.split(',')` with a depth-aware splitter that breaks only on top-level commas, tracking `()`, `[]`, and quote nesting. Recurse into nested declaration blocks, applying the same host-prefix + donut `:not(...)` exclusion to nested selectors. Add `css.test.ts` fixtures for `:is()/:where()/:not()` commas, `[data-x="a,b"]`, `:nth-child(2 of .a, .b)`, and `&`-nesting.

- [ ] **[MEDIUM] FW301 server-fact-in-local-state uses string-prefix name matching** — see Architectural shortcuts above (`component-contracts.ts:36-61,282-288`). Listed there as the primary entry; the substitution of a string-prefix proxy for a dataflow fact is the same root cause flagged in this category.

### Faked v1 behavior

- [ ] **[CRITICAL] FW201 unserializable-capture check is an 8-name denylist, not the exhaustive capture-channel test §4.3 requires** — `packages/compiler/src/lower/handlers.ts:225-230` (SPEC §4.3 — capture channels exhaustive, "Anything else is FW201")
  - **What:** `capturesUnserializableReferences` hardcodes `['window','document','db','request','response','Date','Map','Set'].some(name => referenceSet.has(name))`. The FW201 help text (`@jiso/core/diagnostics.ts:108-111`) and `handler-lowering.test.ts:157-159` literally advertise/assert this 8-name list.
  - **Why it's a hack:** §4.3 defines the capture channels (ctx/query state, `data-p-*` element params, module scope) as exhaustive and makes EVERYTHING else FW201 — a principled check is an allowlist (each free identifier must be a proven channel member). This denylist passes through every non-listed free capture (`fetch`, `localStorage`, `navigator`, a captured local-closure variable, a non-const import, a captured DOM ref), lowering it to a handler that throws or misbehaves on the client. FW201 is asserted DONE but only fires for curated browser globals; the parser already excludes locally-declared names (`parse.ts:1690`), so `references` is a clean free-identifier set the denylist leaks.
  - **Fix:** Invert to channel membership. For each free identifier, emit FW201 unless it is (1) ctx/state-rooted, (2) already captured as a serializable `data-p-*` element param, or (3) a proven serializable/shared module-scope binding. Update the FW201 help text and `handler-lowering.test.ts:157-159` to stop advertising the denylist, and add tests that FW201 fires for `fetch`/`localStorage`/a captured outer-closure local so the curated behavior cannot regress.

- [ ] **[CRITICAL] Platform substitution pattern-matches the method name only; it never proves handler≡platform feature** — `packages/compiler/src/lower/platform.ts:45-103`, decision in `packages/compiler/src/scan/parse.ts:678-741` (SPEC §5.2 rule 4)
  - **What:** `platformSubstitutionFor` branches solely on `method === 'showModal'|'close'|'requestClose'|'togglePopover'…` and returns a substitution carrying whatever host `tag`/`target` it was handed. `<span onClick={() => document.getElementById('x')!.showModal()}>` yields `command="show-modal"` on a `<span>`; `togglePopover()` on a plain `<div>` emits `popovertarget` — and the working JS handler is dropped.
  - **Why it's a hack:** Invoker commands only fire from a `<button>` targeting the right element kind (`<dialog>` for show-modal, a `popover` element for popover actions). The pass verifies neither the host tag nor the target element type, so it substitutes an inert platform attribute and deletes the working handler — silently breaking the interaction. Rule 4 requires the compiler to PROVE equivalence before dropping the handler; this proves nothing, only recognizes fixture method names.
  - **Fix:** Before returning any dialog/popover substitution, prove from typed model facts that (a) the host tag is a valid invoker host (`button`), and (b) the resolved target id (via the component model's id registry that FW221 uses, not the raw `getElementById` string) refers to the required kind — `<dialog>` for show-modal/close/request-close, a `popover`-bearing element for hide/show/toggle. On failure return null (preserve the JS handler) and optionally emit a teaching note per rule 5. Add tests for non-button hosts, non-dialog show-modal targets, and popover actions aimed at non-popover elements.

Add an integration test that drives a fragment apply through the actually-installed loader (not a hand-built root) and asserts focus+selection survive.

- [ ] **[HIGH] FW235 (hand-authored string-rendered component) is gated on an HTML-tag-sniffing regex; tagless string renders escape** — `packages/compiler/src/validate/authoring-surface.ts:51-66`, gate from `firstHtmlTagNameFromLiteralText` at `packages/compiler/src/scan/parse.ts:941-943` (SPEC §5.2 rule 7 / FW235)

- [ ] **[HIGH] Dynamic-valued `viewTransitionName` is silently dropped — no stamp, no diagnostic, raw JSX leaks into HTML** — `packages/compiler/src/lower/view-transitions.ts:18-30` (SPEC §6.4 / §4.2 cross-document view transitions)

## Prioritized sequencing

1. **FW201 channel-membership inversion + platform-substitution equivalence proof** (both CRITICAL, `lower/handlers.ts`, `lower/platform.ts`). These are the two compile-time _safety_ guarantees that silently ship broken/unsafe output today (unserializable closures reaching the client; working JS handlers deleted in favor of inert platform attributes). They are self-contained, do not depend on other work, and close the highest-blast-radius correctness holes.
2. **Wire the real DOM morph into all three production seams, then implement the DOM-backed template-stamp reconciler** (both CRITICAL, runtime). The morph fix is pure wiring of code that already exists and is the single most-leveraged change — it makes §9.1 survival real for every shipped app and lets the morph/fragment-apply tests actually exercise the production path. The template-stamp reconciler should follow immediately and reuse the morph's `domMorphKey` helper, so the two share one fw-key identity contract per §13.2.
3. **Fix the §4.9 coverage holes: compound-query positions and the missing `fragment` status** (both HIGH, `analyze/query-updates.ts`). These restore the completeness proof's soundness — the first stops silent staleness escaping the proof, the second stops false-positive build-blocking FW311 on correctly-declared fragmentTargets. Small, localized, and they unblock authors immediately. Bundle the multi-attribute inline-derive fix (LOW) here since it shares the same file and clears the same spurious FW311s.
4. **Replace the heuristic-for-proof gates with typed/dataflow checks** (FW302 state-shape, FW301 dataflow, FW320 provenance, FW221 per-component scope, Drizzle db-type provenance — HIGH/MEDIUM). Sequence FW302 and the Drizzle provenance gate first (real soundness holes), then the lint-precision fixes. These share the pattern (carry the typed fact instead of matching a name/string) and several reuse existing parser facts, so doing them together amortizes the parser/extraction work.
5. **Make the render-equivalence gate a real differential and wire FW228 into a blocking pipeline** (HIGH). Both are verification gates that currently provide false confidence; fixing them protects every subsequent change from regressing behind a green-but-meaningless check.
6. **Build the §4.6 attribute-merge engine** (HIGH test-gaming, plus it unblocks the FW231/232/233 and FW232-override findings). This is the largest single piece and is already plan-tracked as `fix-ui.md` Phase 2; it should land as a coherent slice that deletes the gallery oracle and drives both the fixtures and the merge diagnostics from one shared compiler module. Schedule after the safety/coverage criticals since it is broad and not on the silent-correctness critical path.
7. **Remaining MEDIUM/LOW Drizzle precision, CSS fallback, mutation-input threading, and hygiene cleanups.** Independent, low-risk, and parallelizable across sub-agents; do them opportunistically once the higher-leverage slices are integrated.
