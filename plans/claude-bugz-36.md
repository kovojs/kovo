## Issues

- [ ] **B1 — A node-postgres `Submittable` (a query value with a `submit()` method) bypasses the DEC-B statement chokepoint entirely on external Postgres: the framework scans `.text`, but node-pg calls `submit(connection)` and sends its own protocol messages, so an app-authored carrier executes `SET kovo.principal='victim'` (or any statement) and forges the principal → cross-owner IDOR read+write on a green build.** (HIGH, framework/security, architectural; `pg-capability-carrier` cap-1; reproduced, paranoid-confirmed)
  - Observed: on external node-pg (not PGlite), an app SQL value implementing the node-pg `Submittable` contract (`{ submit(connection) {…} }`) passed to the managed `db.query(...)`/`.execute(...)` is handed to `client.query()`, which — per node-pg — detects `typeof config.submit === 'function'` and invokes it directly, NEVER routing through the framework's statement scanner (`scanAppPostgresStatementShape`, `managed-db.ts:1189-1314`). The Submittable emits `SET kovo.principal='victim'` (or a multi-statement) at the protocol level; the follow-up read returns the victim's owner-scoped rows.
  - Root cause: the DEC-B chokepoint validates the SQL as a STRING (`.text`) but the driver's execution path is polymorphic — a `Submittable` object routes around the string entirely. The chokepoint is a necessary-but-insufficient check on `.text`; it does not constrain the VALUE's type. followup-11's carrier-snapshot immutability addressed a mutating `.text`, but a `submit()` value never presents a `.text` the framework can trust. The sole-door claim (DEC-B: "app SQL is confined by the chokepoint") is falsified at the driver interface.
  - Why it matters: the entire `kovo.principal` boundary rests on the DEC-B chokepoint being the sole door (the `kovo.role` escalation is closed, but `SET kovo.principal` still impersonates). A `Submittable` is a documented node-pg extension point (pg-cursor/pg-query-stream use it); an authenticated attacker who can get such a value into any managed query path forges any owner id and reads/writes their rows — including `account.password`/`session.token`. It is external-Postgres-specific (PGlite lacks the same Submittable dispatch), i.e. the SUPPORTED production driver.
  - Repro evidence: `/Users/mini/kovo-dogfood-round15/pg-capability-carrier` — a Submittable carrier through the managed handle on node-pg executes an out-of-band `SET` and reads cross-owner. Root: `managed-db.ts` query path passes the value to `client.query()` without rejecting non-plain-object/`submit`-bearing values; `scanAppPostgresStatementShape` only sees `.text`.
  - Acceptance: the chokepoint rejects any query value that is not a framework-constructed plain `{ text, values }` (reject any value with a `submit` function, or reconstruct a fresh plain carrier from the validated text+params and pass ONLY that to the driver, never the app-provided object). A test: a `Submittable`/`submit()`-bearing value through every managed path (query/mutation/task/webhook/escape) is REJECTED, and an out-of-band `SET kovo.principal` cannot execute.

- [ ] **B2 — An INSTEAD OF trigger on a reachable VIEW escapes the DEC-D attached-code audit: `checkPostgresAppDbPosture` stays GREEN while an app-role INSERT/UPDATE/DELETE on the view fires a SECURITY DEFINER trigger function.** (HIGH, framework/security, architectural; `pg-attached-code-completeness` attached-code-view-instead-of-trigger; reproduced, paranoid-confirmed)
  - Observed: a VIEW the app role can write (the reachability query includes `relkind 'v'`, `postgres-runtime.ts:62-68`) carries an INSTEAD OF trigger whose function is `SECURITY DEFINER`; a normal app-role write to the view fires the definer function (exfiltrating / cross-owner-writing), and `checkPostgresAppDbPosture` returns ok=true.
  - Root cause (self-verified): the DEC-D attached-code audit's `pg_trigger` scan (`postgres-runtime.ts:1149-1157`) joins the triggering relation but the no-unexpected-attached-code refusal is not applied to INSTEAD OF triggers on VIEWS the way it is to DML triggers on tables — the view-write side-effect path is outside the modeled set. DEC-D named `{DML trigger, rule, CHECK, default, index-expr}` but INSTEAD OF triggers on views are a distinct trigger class (fired by view DML, `tgtype` INSTEAD OF) that the audit does not refuse.
  - Why it matters: followup-10 DEC-D's whole claim is "no unexpected attached code on app-reachable objects, fail closed on unmodeled mechanisms." An INSTEAD OF view trigger is exactly an unmodeled side-effect mechanism that reaches definer code on an app-role write — the C7/C8 execution-axis edge reproduced one relkind over (round-14 B3 was table triggers; this is view triggers).
  - Repro evidence: `/Users/mini/kovo-dogfood-round15/pg-attached-code-completeness`; a definer INSTEAD OF trigger on a writable view passes `checkPostgresAppDbPosture` and fires on app-role write. Root: `postgres-runtime.ts:1149-1157` trigger scan + the reachable-view handling.
  - Acceptance: the attached-code audit refuses an INSTEAD OF (view) trigger whose function is not framework-vetted, exactly as it refuses a DML (table) trigger; add the view-trigger + constraint-trigger classes to the DEC-D coverage and the DEC-F fuzzer mechanism axis. A test: a definer INSTEAD OF trigger on a reachable view makes `kovo db check` REFUSE.

- [ ] **B3 — An ADOPTED `kovo_reader`/`kovo_writer` role that a DBA pre-created WITH `BYPASSRLS` passes provision and `kovo db check` GREEN, but the runtime assumes it via `SET ROLE`, silently disabling ALL owner-scope RLS → total cross-owner read+write.** (HIGH, framework/security, architectural; `pg-role-topology-provision` pg-topo-adopted-reader-writer-bypassrls-unchecked; reproduced, paranoid-confirmed)
  - Observed: with `KOVO_DB_READER_ROLE`/`KOVO_DB_WRITER_ROLE` pointing at DBA-created roles that happen to have `BYPASSRLS`, provision and `kovo db check` report OK; at runtime the login `SET ROLE`s to the adopted reader/writer, and because that role has `BYPASSRLS`, `FORCE ROW LEVEL SECURITY` is ignored — every owner table is fully cross-owner readable/writable.
  - Root cause (self-verified): the adopted-role validation (`postgres-runtime.ts:2395-2407`) checks only that adopted roles EXIST (`missing` set); it never checks their `rolbypassrls`/`rolsuper`. The least-privilege posture check `runtimeConnectionLeastPrivilegeIssue` (`:2610-2650`) validates `current_user` — the LOGIN role's `rolbypassrls`/membership — but NOT the attributes of the reader/writer roles the login ASSUMES. So a `BYPASSRLS` adopted role sails through both gates. The identity invariant (followup-9 B2) reasoned about the login role and missed the assumed roles.
  - Why it matters: the adopted-role path (postgres-v1-devex DEC-B2, for locked-down/enterprise providers) is a supported production deploy; a single DBA-side role attribute (`BYPASSRLS`, easy to set by habit) disables the ENTIRE engine-choke owner-scoping guarantee while every Kovo gate says green. This is the identity-axis edge: the check covers the role you authenticate as, not the roles you can become.
  - Repro evidence: `/Users/mini/kovo-dogfood-round15/pg-role-topology-provision`; an adopted `BYPASSRLS` reader passes `kovo db check` and serves cross-owner. Root: `postgres-runtime.ts:2395-2407` (existence-only adopted validation), `:2610-2650` (login-role-only least-priv check).
  - Acceptance: provision + `kovo db check` + boot verify that EVERY app role the runtime can assume (reader/writer, and admin/system if reachable) is `NOBYPASSRLS` and `NOSUPERUSER`; an adopted `BYPASSRLS` reader/writer makes provision/check REFUSE. A test: a `BYPASSRLS` adopted role fails the posture gate.

- [ ] **B4 — REGRESSION of `bugz-24` A1: the Better Auth adapter reads `session.token` / `account.password` / `accessToken` / `refreshToken` UNBOXED through the framework `systemDb` capability, so auth secret columns are handled as plain strings outside the readonly secret-boxing choke.** (MED, framework/security; `pg-auth-deep` pg-auth-deep-1; reproduced, paranoid-confirmed, verifier-corrected HIGH→MED)
  - Observed: followup-11 removed the app-EXPORTED `appRuntimeAuthDb`, but the auth adapter (`auth.$context.adapter` / `createAuthAdapter`) reads the credential columns via the privileged `systemDb` path unboxed; the secret-boxing read boundary (which throws on wire coercion) is not on the adapter's read path.
  - Root cause: the capability was removed as an app EXPORT but the adapter — the internal consumer — still reads secrets as ordinary values through `systemDb` (`postgres-runtime.ts:433/572`). followup-11's principle ("capabilities are not values") narrowed the export surface but the adapter itself materializes the secret as a plain string; if any adapter-configured hook, log, or error path observes it, it leaks. Verifier corrected HIGH→MED because reachability is the adapter's internal read (Better Auth needs the hash/token to validate), not a direct app read — but it is unboxed and thus not proven confined.
  - Why it matters: `bugz-24` A1's acceptance was that auth secrets never cross a boundary as plain strings; the adapter path still does. Auth credentials are the highest-value secret; "the boxing choke never sees the adapter read" is the exact A1 finding, reduced but not eliminated.
  - Repro evidence: `/Users/mini/kovo-dogfood-round15/pg-auth-deep`; the adapter read returns unboxed `session.token`/`account.password`. Root: the auth adapter's `systemDb` read path bypasses `createSecretBoxingReadDb`.
  - Acceptance: the auth adapter reads credential columns through a boxing/least-exposure path (or a documented, audited, log-safe internal channel that provably cannot reach a wire/log/error), so a boxed/confined value — not a plain string — crosses every boundary; a test asserts the adapter read is confined. Pairs with `papercuts-34` P2 (kovo_writer over-privileged on secret columns).
