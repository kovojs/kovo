# Bugz 21

Created 2026-06-30. Source of truth remains `SPEC.md`. Five confirmed security/soundness defects
escalated from the postgres-template dogfood sweep (companion papercuts ledger:
`plans/papercuts-super-10.md`), reproduced against a clean `create-kovo` (default postgres) **prod
build** on `main`. B1 (SSR XSS) was self-verified first-hand end-to-end; the rest are source-confirmed
with track-verifier reproductions.

**Two meta-threads.** (1) **Validation gates are keyed on syntactic accidents, not provenance** — the
authoring-surface validator only runs on files containing the substring `component(` (→ B1 SSR XSS),
and the raw-SQL trust waiver matches the callee _name_ `trustedSql` rather than its import (→ B2 gate
disable). CLAUDE.md explicitly forbids name-/substring-based brands as security shortcuts. (2) **The
`bugz-20` B1 raw-SQL fix has three escape hatches**: the static gate is name-shadow-bypassable (B2),
`trustedSql` waives it with any non-empty string (`papercuts-super-10` notes), and the SPEC-mandated
**runtime** `tables:` enforcement was never shipped to production (B5) — so a `trustedSql` write whose
declared tables drift from what it mutates has no backstop.

## Scope

- Apps: fresh `create-kovo` default (postgres/PGlite) scaffolds, link-local to the monorepo, driven
  through `build:prod` → `NODE_ENV=production node dist/server/server.mjs` → real browser/curl.
- Out of scope: the `<Defer>`-nesting / slowloris / bulk-serverValue / no-logging papercuts (in
  `papercuts-super-10.md`).

## Issues

- [ ] **B1 — SSR XSS: the authoring-surface validator (incl. the KV235 internal-import gate) only runs on files whose text contains `component(`, so an ordinary app `.ts` helper can statically import the `@internal` raw-HTML sink `renderedHtml` and emit attacker-shaped HTML unescaped — through a green `kovo build`.** (HIGH, framework; found by `t5-type-boundary`; self-verified)
  - Exploit / observed: a plain module `src/raw-helper.ts` = `import { renderedHtml } from '@kovojs/server/internal/html'; export const rawUnescaped = (m) => renderedHtml(m)` (the file does **not** contain the substring `component(`) builds with **exit 0 and zero KV235**. A component rendering `{rawUnescaped('<img src=x onerror="alert(1)"><b id="xss-probe">RAW</b>')}` serves the markup **verbatim/unescaped** in the authed `GET /` HTML. Placing the _same_ import in a file that happens to contain `component(` _does_ throw KV235 — that substring is the entire difference. The `trustedHtml`/KV236/KV426 escaping-audit boundary is bypassed.
  - Root cause: `packages/compiler/src/vite.ts:551` — `shouldTransformViteComponentSource` does `if (!source.includes('component('))  return false;`, gating **all** authoring-surface validation (including the KV235 `@internal`-import gate in `validate/authoring-surface`) behind that literal substring; the standalone path (`vite.ts:304-311`) runs only registry lowering and never calls `validateAuthoringSurface`. `renderedHtml` is honored by the server renderer with zero escaping.
  - Why it matters: SSR XSS is the highest-severity web defect, and KV235/the escaped-HTML brand are the framework's "secure by construction" headline. The gate is bypassed by a syntactic accident — any helper module, any agent-generated util, any file that doesn't literally contain `component(` — and ships through the deploy gate with no diagnostic. CLAUDE.md: brands/gates "should either route through a validating constructor or make an already-enforced invariant clearer" — never a substring shortcut.
  - Repro evidence (self-verified, clean base scaffold, prod): `raw-helper.ts` importing `renderedHtml` (no `component(`), rendered in `ContactsRegion` → `build:prod` exit 0, no KV235; prod `GET /` served HTML contains `<img src=x onerror="alert(1)"><b id="xss-probe">RAW</b>` raw (1 occurrence, **0** escaped `&lt;img`). Source: `vite.ts:551`.
  - Fix / acceptance: run the authoring-surface validator (KV235 internal-import gate) on **every** authored module regardless of whether its text contains `component(` — key on the real authoring surface / import provenance, not a substring. Prove with a build test: an app `.ts` module importing `@kovojs/server/internal/html` fails KV235, and an end-to-end prod-artifact test that the served HTML escapes attacker HTML. (Same name-/substring-not-provenance class as B2.)

- [ ] **B2 — The raw-SQL KV414/KV438 trust waiver matches the callee _name_ `trustedSql`, not its import provenance, so a local `function trustedSql(s){return s}` shadow disables the IDOR + mass-assignment gates and ships green — a bypass of the `bugz-20` B1 fix.** (HIGH, framework; found by `t2-rawsql-depth`)
  - Exploit / observed: an owner-table raw write `request.db.execute(sql\`UPDATE notes … WHERE id=${id}\`)`correctly fires`KV414`(IDOR) +`KV438`(mass-assignment) → red build. Wrapping the *same* statement in a **local identity function** named`trustedSql` (`function trustedSql<T>(s,\_o){return s}`— not the`@kovojs/drizzle`constructor) makes`build:prod`go **fully green** with zero KV414/KV438/KV422. The marquee write-side gates are silently disabled by a name collision, and`kovo explain --writes` records a fake "audited escape" that was never the real attested brand.
  - Root cause: `packages/drizzle/src/static.ts:2154-2163` `isTrustedSqlArgument` returns `callee.getText() === 'trustedSql'` (and `staticAccessName(callee) === 'trustedSql'`) — **no `@kovojs/drizzle` symbol/import resolution**; also `:1417` (sqlTextSafety) and `:1959-1960` (`rawWriteSqlScopeFacts` short-circuits to empty when `trust.trusted`). Any same-named identifier in scope satisfies it.
  - Why it matters: `trustedSql` is the _only_ shippable raw-write path (every non-trusted raw write trips KV438 on the auto-governed PK), so the trust check is load-bearing for the entire raw-SQL security story — and it's defeated by an identifier name. CLAUDE.md explicitly forbids this: "Do not use casts, public structural brand fields, or `Symbol.for()` brands as shortcuts around a security invariant." A name-based match is the same anti-pattern.
  - Repro evidence: `t2-rawsql-depth` probe `pShadowTrusted` — local `trustedSql` identity wrapping an owner-table `db.execute` write → `build:prod` GREEN (no KV414/KV438); the same write without the shadow → KV414+KV438 red. Source: `static.ts:2154-2163`.
  - Fix / acceptance: resolve `trustedSql` to the `@kovojs/drizzle` export by symbol/import provenance (per SPEC §11.1) before honoring the waiver; a same-named local identifier must not suppress the gates. Prove with a build test: a local `trustedSql` shadow still fires KV414/KV438.

- [ ] **B3 — Mutation handlers run with NO framework transaction by default: a thrown / constraint-failing multi-step handler leaves a partial write (SPEC §10.3 BEGIN/COMMIT lifecycle unimplemented).** (HIGH, framework; found by `t6-transactions`)
  - Exploit / observed: a handler `request.db.insert(contacts).values({id:'partial-A',…})` then `throw new Error('boom')` returns HTTP 500, and the DB afterward shows `partial-A` **persisted** — the pre-throw write is committed. Reproduced deterministically. An explicitly `request.db.transaction(...)`-wrapped handler _does_ roll back. So atomicity is purely opt-in.
  - Root cause: `packages/server/src/mutation.ts:267-269` — `value = definition.transaction ? await definition.transaction(lifecycleRequest, runHandler) : await runHandler(guardedRequest)`: the framework opens **no** transaction; the handler runs directly on the plain read-write `request.db`. SPEC §10.3 describes a `BEGIN`/`COMMIT` mutation lifecycle (a handler error rolls back), which is not implemented for the default handler form.
  - Why it matters: "a mutation is atomic" is a baseline data-integrity expectation and the SPEC §10.3 contract. The default, idiomatic `mutation({ handler })` — the shape the starter ships and every author copies — is **not** atomic: any multi-step write that fails partway corrupts data (half-created orders, money moved but not recorded), and there is no diagnostic steering authors to wrap it.
  - Repro evidence: `t6-transactions` — `POST /_m/probes/no-tx-partial` → 500, `/api/dbstate` shows `partial-A` present (ports 7610 & 7640); the `transaction(...)` probe rolled back. Source: `mutation.ts:267-269`.
  - Acceptance: the default mutation handler runs inside a framework-opened transaction that rolls back on a thrown handler / DB error (or the build/diagnostic forces an explicit transaction for multi-write handlers). Prove with an end-to-end prod-artifact test: a handler that writes then throws leaves the DB unchanged.

- [ ] **B4 — The pre-dispatch per-IP rate limiter collapses to one shared `'unknown'` bucket in the default node deployment, so any anonymous client can 429 ALL mutations app-wide with ~120 cheap requests (DoS).** (HIGH, framework; found by `t4-rate-limit`)
  - Exploit / observed: in the default node prod artifact, `requestClientIp` returns `undefined` (because `trustedProxy` defaults `false`), and the limiter keys on `ip ?? 'unknown'` — so **every** anonymous request shares a single `'unknown'` bucket. ~120 cheap requests from one client exhaust the shared per-IP mutation budget and every other user's mutations then 429 app-wide.
  - Root cause: `packages/server/src/app-load-shed.ts:455` `requestClientIp` returns `undefined` when `!trustedProxy` (`:114` default false / `:89`), `:216` keys the bucket on `ip ?? 'unknown'`, `:60-64` per-IP mutation max 120. The default node preset sets no `clientIp` extractor, so the per-IP limiter degenerates into a single global bucket.
  - Why it matters: a rate limiter that lumps all anonymous traffic into one bucket converts a protection into a self-inflicted DoS amplifier — one attacker (or one busy NAT'd client) trivially denies mutations to everyone. The default deploy posture (no `trustedProxy`, no `clientIp`) is exactly what the starter ships.
  - Repro evidence: `t4-rate-limit` — ~120 anonymous requests trip the shared `'unknown'` bucket → subsequent mutations from any client 429. Source: `app-load-shed.ts:216,454-455,114,60-64`.
  - Acceptance: the default deployment derives a real per-connection client key (e.g. socket remote address) for the rate-limit bucket, or disables per-IP limiting rather than collapsing to a shared bucket; document the `trustedProxy`/`clientIp` requirement. Prove with a test: requests from distinct peers get distinct buckets in the default node preset.

- [ ] **B5 — The production executor never parses/enforces the declared raw-SQL `tables:` allowlist (SPEC §10.3:1167/§11.2 runtime fail-closed is absent); the only owner-write runtime check lives in `@kovojs/test`. The `bugz-20` B1 Phase 3 close rests on test-harness evidence, not the deployed artifact.** (MED, framework; found by `t1-regression-sweep`; runtime half of `bugz-20` B1)
  - Observed: the production managed db handle (`packages/server/src/sql-safe-handle.ts:153-207` `assertManagedSqlStatement`) wraps query/exec/execute only to assert the Kovo SQL brand (KV422) — it never parses the executed statement's tables against the registry-declared `tables:`. There is no table-allowlist fail-closed and no conservative-invalidation-on-undeclared-table in the request path. `pgsql-ast-parser` and `assertOwnerWritesScoped` (`packages/test/src/verifier.ts:372`) ship **only** in `@kovojs/test` (an author-opt-in test facility), never in `dist/server`.
  - Root cause: `sql-safe-handle.ts:153-207` is the entire production managed-SQL floor (brand-only, `void mode`); the `tables:` field (`packages/core/src/graph.ts:43`) is consumed only by `@kovojs/test` (`verifier-sql.ts:228`, `verifier-diagnostics.ts:199`), with zero consumers in `packages/server/src`. SPEC §10.3:1167/§11.2 is normative that the **executor** "parses and enforces" `tables:` and "in production … fails closed … records the violation."
  - Why it matters: because `trustedSql` suppresses the static gates (and the justification content is unvalidated), the runtime `tables:` allowlist is the _only_ backstop for a trusted statement whose declared tables drift from what it actually mutates (e.g. `tables:['contacts']` but a conditional arm writes `account`/`user`). That backstop does not run in the deployed artifact — so a `trustedSql` write to an undeclared owner/credential table is unaudited at build (waived) **and** at runtime (no enforcement). Combined with B2, the raw-SQL gate has no sound floor.
  - Repro evidence: source + dependency structure — `pgsql-ast-parser` absent from `@kovojs/server`/`@kovojs/core` deps and from the built dogfood app; `tables:` consumed only by `@kovojs/test`; `bugz-20` B1 Phase 3's runtime evidence is `pglite-harness.test.ts` (the test harness). Source: `sql-safe-handle.ts:153-207`, `core/src/graph.ts:43`.
  - Acceptance: the production executor parses each raw-SQL write's mutated tables and fails closed (conservative whole-domain invalidation + recorded violation) when it touches a table outside the declared `tables:`, per SPEC §11.2 — wired into `dist/server`, not only `@kovojs/test`. Re-open `bugz-20` B1 Phase 3 with an end-to-end prod-artifact assertion (a `trustedSql` write whose runtime tables drift is caught), not a test-harness one.
