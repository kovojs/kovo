# Round-10 Soundness Bugz 31

Created 2026-07-02. Source of truth remains `SPEC.md`. Security/soundness fail-opens found dogfooding AFTER
`plans/fundamental-fixes-followup-5.md` (authorization moved to runtime chokes — Postgres RLS, SQLite predicate
injection, DEC-K census, actAs/declareSystem, ownerVia) landed on main. All reproduced under `KOVO_PARANOID=1` with a
served cross-owner leak on the prod artifact. Over-blocks + ergonomics are in `plans/claude-papercuts-29.md`.

**Meta-theme — followup-5 moved owner authorization to a runtime choke, but on SQLite that choke is a Drizzle
_builder-shape proxy_, not a connection/statement-level door, so it is architecturally incapable of being the _sole_
door.** The Postgres side puts the owner predicate at the connection (`FORCE ROW LEVEL SECURITY` + non-superuser
`kovo_reader` + unset `kovo.principal` ⇒ default-deny), so _every_ read through that connection — builder, relational,
raw, ambient singleton — fails closed by construction. The SQLite side puts the owner predicate in
`createSqliteAuthorizationDb`, a `Proxy` get-trap that rewrites `config.where` on the builder verbs it recognizes
(`managed-db.ts:861-905`). Two reachable read paths escape that proxy entirely, and on SQLite (no engine RLS) the
proxy is the _sole_ choke, so each escape is a true fail-open — while the identical code fails closed on Postgres:

1. **A sibling read handle the proxy never wraps** — the exported `readonlyAppDb` module singleton (B1).
2. **An object-valued property the proxy returns raw** — the `db.query` relational namespace (B2).

This is the round-1..9 pattern once more, now on authorization's runtime choke: enforcement sits at a choke, but the
choke is not confined to a sole door — the object graph has other reachable read shapes the interceptor doesn't cover.
Rice again: you cannot proxy-intercept every reachable read shape on a mutable object graph. SQLite owner-scoping must
move to a door that covers the driver/statement layer regardless of builder shape (fail closed at `.prepare` for any
owner-table read lacking a proven principal predicate), the way PG's connection-level RLS already does.

Baseline: fresh `--sqlite` + `--dialect postgres` starters linked to local framework, built + served with
`KOVO_PARANOID=1`. Each finding independently reproduced by a skeptical verifier; the two HIGH roots re-verified
first-hand in framework source (below).

## Issues

- [ ] **B1 — The exported `readonlyAppDb` SQLite read singleton is NOT owner-scoped: a public endpoint (or any user-authored read) that imports it — exactly as the template documents — serves every owner's rows cross-owner on a green `KOVO_PARANOID=1` build. Postgres fails closed.** (HIGH, framework/security, architectural; converged from 3 independent tracks — `census-completeness`, `ingress-endpoints-tasks-webhooks`, `escapes-rawread-allowlist`; reproduced, paranoid-confirmed)
  - Observed: an owner table `notes`/`orders`/`secrets_notes` (`kovo({owner:'userId'})`) seeded with two owners' rows. A public `GET` endpoint reads it via `readonlyAppDb` and returns BOTH owners' rows including per-owner secret columns (`{"rows":[{userId:owner-a,body:A-PRIVATE},{userId:owner-b,body:B-SECRET}]}`, HTTP 200) with no session/principal. In the SAME app the sanctioned scoped paths fail closed correctly: `ctx.actAs('owner-a').db.read` → only owner-a's row; the request-threaded `context.db` in a query loader → only the caller's row. So it is specifically the `readonlyAppDb` handle that leaks. The identical source on `--dialect postgres` returns `{"rows":[]}` (RLS default-deny). Green `KOVO_PARANOID=1 pnpm run build:prod` (exit 0), no KV414 / census diagnostic.
  - Root cause (self-verified): `packages/create-kovo/templates/src/_kovo/app-runtime-db.sqlite.ts:110` binds `readonlyDb: secretReadDb`, where `secretReadDb = createSecretBoxingReadDb(readonlyDb(db, …))` (lines 73-89) — **secret-boxing only, never wrapped in `createSqliteAuthorizationDb`**; `:141` re-exports it verbatim as the module-level, principal-less singleton `appRuntimeReadonlyDb`. The owner predicate is applied _only_ inside `appRuntimeDbProvider(request)` (`:149`, `:157`, `:165` — request-scoped), reachable from an endpoint solely via `ctx.actAs(id)`. `packages/create-kovo/templates/src/db.ts` re-exports it as `readonlyAppDb` with the doc _"The running app read surface. Endpoint/user-authored reads should import this value"_ — so **the leaking path is the blessed/idiomatic one, not author misuse.** SQLite has no engine RLS, so the un-wrapped singleton skips owner-scoping AND the DEC-K census (`createAuthorizationCensusDb`) entirely and reads raw. Contrast Postgres `app-runtime-db.ts`: `readonlyDb = createRequestScopedReadonlyDb(client)` binds a non-superuser role under `FORCE ROW LEVEL SECURITY` with `kovo.principal` unset ⇒ default-deny for any principal-less read. The DEC-H sole-door lint `endpointRawDriverImportDiagnostic` (`packages/drizzle/src/static.ts:1124`) is meant to block exactly "a raw handle bypassing the endpoint authorization choke," but its `ENDPOINT_RAW_DRIVER_MODULES` set (`:1117-1122`: `better-sqlite3`, `drizzle-orm/*`, `@electric-sql/pglite`) misses the template's own `./db.js` re-export of the same unscoped handle.
  - Why it matters: **cross-owner/cross-tenant IDOR of secret columns through the exact handle the scaffold tells authors to use for endpoint reads, on a green paranoid build.** followup-5's "sole door" claim for SQLite is false: `createSqliteAuthorizationDb` is one of two exported read doors, and the other (`readonlyAppDb`) has no owner scope. The dialect split (PG closed, SQLite open) is the tell — the guarantee rides PG's connection-level RLS, which SQLite's builder-proxy does not replicate.
  - Repro evidence: `/Users/mini/kovo-dogfood-round10/{census-attack,ingress-sqlite,roleak}` (SQLite) leak; `/Users/mini/kovo-dogfood-round10/{ingress-pg,roleakpg}` (PG) return `[]`. Self-verified in source: `app-runtime-db.sqlite.ts:110/141` (boxing-only singleton, no authz wrapper) vs `:149/157/165` (authz applied only in the request provider).
  - Acceptance: the exported read singleton must be owner-scoped and census-gated for EVERY caller — either (a) `readonlyAppDb` refuses any owner/unclassified-table read that carries no proven principal (fail closed, matching PG default-deny), or (b) SQLite owner-scoping moves to the driver/statement layer so no exported handle can read an owner table unscoped, or (c) the template stops exporting a principal-less read handle and forces all endpoint/user reads through the request-threaded scoped capability (`ctx.actAs`/`context.db`). Reclassify: the SQLite owner-scope choke is only sound if it is the SOLE reachable read door. Add a paranoid test: an endpoint reading an owner table via `readonlyAppDb` must not serve another owner's row.

- [ ] **B2 — The SQLite owner-predicate proxy does not cover the Drizzle relational query builder (`db.query.*.findMany/findFirst`): once an app wires `relations`, every owner-scoped table is fully readable cross-owner on a green `KOVO_PARANOID=1` build. Postgres fails closed.** (HIGH, framework/security, architectural; `sqlite-injection-version-split`; reproduced, paranoid-confirmed)
  - Observed: owner table `orders` (`kovo({owner:'userId'})`), seeded order-a→Alice, order-b→Bob. Signed in as Alice: `GET /_q/queries/orders-plain-query` (builder `.select().from(orders)`) correctly returns ONLY Alice's row (scoped, no over-block); `GET /_q/queries/orders-relational-query` (`db.query.orders.findMany()`) returns BOTH Alice's and Bob's rows including `BOB-SECRET`. Symmetric as Bob. The owner scope is present on the builder path and entirely absent on the relational path.
  - Root cause (self-verified): `packages/server/src/managed-db.ts` `sqliteAuthorizationProxy()` get-trap intercepts only the query-builder _verbs_ — `select`/`selectDistinct` (`:886-893`), `update`/`delete`/`insert` (`:895-897`), the terminals in `SQLITE_AUTHZ_TERMINALS` (`:875-884`) — each of which is a function. **Any non-function property is returned RAW at `:866-867 (`if (typeof item !== 'function') return item;`).** The Drizzle RQBv2 relational namespace `db.query` is an OBJECT, so it is returned completely unproxied; `db.query.orders.findMany()` builds and runs its own SQL via a code path that never consults the injected `config.where`. There is no relational-builder handling anywhere in the SQLite authz path. The read handle actively advertises this surface: `readonlyCapabilityDb()` returns `db.query` raw when non-function (`managed-db.ts:1340-1342`), `query` is in `READ_CAPABILITY_PROPERTIES` (`:260`, not denied), and `Reader<Db>` re-exposes `query` (`:524-530`). Because SQLite has no engine RLS, this proxy IS the sole owner-scope choke, so the miss is a true fail-open. (Contrast: on Postgres, `db.query` reads still run under the RLS connection and default-deny.)
  - Why it matters: same root as B1 — the proxy covers a bounded set of builder verbs but the object graph exposes a whole parallel read API (`db.query`) it never touches, and the `Reader` type advertises it as a capability. A one-property escape reads every owner-scoped table cross-owner.
  - Reachability caveat (bounds but does not nullify the fail-open): the shipped starter's generated `drizzle({ client, schema })` omits `relations`, so out-of-the-box `db.query` is empty. To reach the leak the app must add `defineRelations` and pass `relations` into the `drizzle()` call in `src/_kovo/app-runtime-db.ts` — benign ORM config, not a guard opt-out. Once relations are wired (a first-class Drizzle feature the `Reader` type surfaces), every owner table is readable via `findMany`/`findFirst`.
  - Repro evidence: `/Users/mini/kovo-dogfood-round10/vsplit-app`; `KOVO_PARANOID=1` build exit 0; served `/_q/queries/orders-relational-query` returned ALICE-SECRET AND BOB-SECRET to Alice. Self-verified in source: `managed-db.ts:866-867` returns non-function props raw; `:1340-1342` `readonlyCapabilityDb` returns `db.query` raw; `:524-530` `Reader<Db>` re-exposes `query`.
  - Acceptance: `db.query.*` relational reads of an owner table are owner-scoped by the same choke as the builder (inject the owner predicate into the relational config), OR the framework must move SQLite owner-scoping to the driver/statement layer so the read shape is irrelevant (preferred — see meta-theme; also fixes B1). Add a paranoid test: `db.query.<ownerTable>.findMany()` must not return another owner's row.

- [ ] **B3 — The DEC-K static census reachable-table denominator excludes endpoint/task/webhook handler reads, so the "unclassified table fails the BUILD" half of the census guarantee does not hold for endpoint-only tables.** (MED→LOW alone, HIGH as an amplifier of B1; `census-completeness`; reproduced, fail-closed in isolation)
  - Observed: an unclassified table read ONLY from an endpoint handler (never from a query loader) is NOT flagged by the static census — `KOVO_PARANOID=1 pnpm run build:prod` stays green with zero KV414. The same table read from a query loader IS caught (KV414), and a JOIN-only read into a classified loader is also caught, so joins are in the denominator but endpoint/task/webhook handler bodies are not.
  - Root cause: `packages/drizzle/src/static.ts:2102` `requestReachableTableNames()` sources the census denominator only from query `readProvenance`, `queryWriteReachability`, and the `touchGraph` (mutation/query touches). Endpoint/webhook/task handlers are never analyzed for Drizzle `.from()`/join reads, so tables reachable solely through them are absent from `reachable` and `authzCensusDiagnosticsFromTables` (`static.ts:2148`) never evaluates their classification.
  - Why it matters: in isolation this is fail-CLOSED — an endpoint read through the request-scoped `createSqliteAuthorizationDb` handle (`ctx.actAs`) returns `[]` for an unclassified table (`managed-db.ts:657-676` throws KV414 for raw SQL; builder returns no rows), so the verifier corrected it to PARTIAL/LOW on its own. **But combined with B1 it is the amplifier: the endpoint census gap means nothing warns at build, and `readonlyAppDb` means nothing denies at runtime — together they are the served leak.** The census's own "unclassified ⇒ build error" promise silently does not cover the endpoint/task surface.
  - Acceptance: `requestReachableTableNames()` includes Drizzle reads in endpoint/webhook/task handler bodies in the census denominator (so an unclassified endpoint-only table fails the build), independent of the runtime floor. Pairs with B1 — the runtime read door must also fail closed for unclassified/owner tables.

## Refuted / Not Carried Forward

- **PG runtime RLS + census keyed off hand-maintained `SCHEMA_TABLES` (decoupled from `schema.ts`) — refuted as a fail-open:** `managed-db.ts:665 if (!inSchema) continue;` skips a table absent from census metadata, but the generated `app-runtime-db.ts` keys ALL surfaces (`SCHEMA_DDL`, RLS policies, census metadata) off the SAME `SCHEMA_TABLES` list, so a table the census skips is one that received no `CREATE TABLE` and does not physically exist — reading it errors, it cannot leak. The static KV414 gate reads `schema.ts` directly and catches owner tables at build. Real LOW ergonomic papercut only → `papercuts-29` P4.
- **census-completeness static gap as a standalone served leak — refuted:** the request-scoped runtime floor (`ctx.actAs` → `createSqliteAuthorizationDb`) is fail-closed for unclassified tables; only dangerous combined with B1 (recorded as B3).
