Created 2026-07-03. Source of truth remains `SPEC.md`. Security fail-opens found dogfooding AFTER

**Meta-theme — followup-8's "ask the engine, don't model it" was the right move but asks an INCOMPLETE set of
questions.** The plan's C6 / DEC-A claim (`followup-8.md:33,58`) that `has_table_privilege` "resolves PUBLIC, role
membership, COLUMN GRANTS, and default privileges by construction" is factually FALSE: `has_table_privilege` is
strictly TABLE-level and returns `false` for a column-only grant. The audit asks the engine, but with the WRONG
GRANULARITY (table, not column), a LITERAL grantee predicate (not effective privilege), and a RESTRICTED schema scope
(routines only in app schemas). Each gap is a cross-owner/secret fail-open on a GREEN `checkPostgresAppDbPosture`:

| Audit query | Granularity/predicate | Misses (→ fail-open) |
| --- | --- | --- |
| `has_table_privilege(role, oid, 'SELECT')` (`:793`) | TABLE-level | column-only grants (B1); sequences (B4) |
| KV435 `column_privileges … grantee = $readerRole` (`:735`) | LITERAL grantee | `GRANT … TO PUBLIC`/writer/membership (B2) |
| routine scan `WHERE nspname IN (app schemas)` (`:989`) | app schemas only | cross-schema SECURITY DEFINER functions (B3) |

The arc's meta-bug, one granularity deeper: the engine oracle is complete ONLY if you ask it at the finest granularity
(per-column, `has_column_privilege`/`aclexplode` over `pg_attribute`), across ALL schemas, over ALL object types
(relations + routines + sequences). The relation scan already uses all-schema (`nspname NOT IN (pg_catalog,
information_schema)`, `:799`) — the routine scan and the column/secret checks did not follow suit. Separately, one auth
consumer skips the `NonRequestPrincipalPosture` brand check (B5).

Baseline: default Postgres/PGlite at `origin/main`, exercised against the REAL boot/`kovo db check` gate
(`checkPostgresAppDbPosture`). B1 self-verified first-hand on raw PGlite; B2/B3/B5 roots self-verified in source; each
independently reproduced by a skeptical verifier. **followup-8's round-12 fixes HELD** (matview / PUBLIC-table /
definer-view / reference-membership now refuse — see Refuted); these are the NEXT doors.

## Issues

- [ ] **B1 — Column-level grants are invisible to the `has_table_privilege` closure audit: a relation reachable ONLY via a column grant (`GRANT SELECT (col) ON t TO kovo_reader`/`TO PUBLIC`) has `can_select=false`, is dropped from the reachable set, and leaks that column cross-owner on a GREEN audit.** (HIGH, framework/security, architectural; converged from `pg-closure-audit-round2` PGCA-1 + `pg-provision-migrate-security` PMS-2 + `pg-realistic-multitenant-round2` COL-GRANT-BYPASS; self-verified first-hand + engine-mechanism reproduced)
  - Observed: a non-Kovo table `billing(customer, card_pan, balance)` with NO RLS, `GRANT SELECT (card_pan) ON billing TO kovo_reader` (column-only). The verbatim audit enumeration (`postgres-runtime.ts:793-796`) returns `has_table_privilege SELECT/INSERT/UPDATE/DELETE = false` for every app role → `billing` is not in the reachable set → the `KV433_REACHABLE_TABLE` catch-all never fires → `ok=true`. Then `SET ROLE kovo_reader; SELECT card_pan FROM billing` returns ALL customers' PANs. COL-GRANT-BYPASS also shows a column INSERT/UPDATE grant → cross-owner WRITE.
  - Root cause (self-verified): `postgres-runtime.ts:793-796` gates reachability on `has_table_privilege`, which is TABLE-level and does NOT reflect column ACLs (a column-only grant returns `false`). `reachableRelationsFromRows` drops relations with `privileges.length === 0` (`:925`), killing the `KV433_REACHABLE_TABLE` backstop (`:835`) for column-granted objects. `has_column_privilege` is used 0 times in the file. First-hand probe (`scratchpad/col-priv-probe.mjs`): `has_table_privilege(reader, side_tbl, 'SELECT')=false` while `has_column_privilege(reader, side_tbl, 'secret', 'SELECT')=true` and `SET ROLE kovo_reader; SELECT secret FROM side_tbl` returned `COLUMN-ONLY-SECRET`.
  - Why it matters: a cross-owner/cross-tenant read (and write) of arbitrary column data (PII, PANs, secrets) on a GREEN `checkPostgresAppDbPosture` — the sole enforcement choke followup-8 declares "the end of the line for auditing the reachable set." A column GRANT (a normal reporting/analytics pattern, via `seedSql` or a hand migration) silently defeats the audit. It directly falsifies the C6 "column grants by construction" claim.
  - Repro evidence: `scratchpad/col-priv-probe.mjs`; source `postgres-runtime.ts:793-796`, `:925`, dead catch-all `:835`.
  - Acceptance: the reachability audit computes effective privilege at COLUMN granularity — enumerate `pg_attribute` per relation and test `has_column_privilege(role, oid, attname, 'SELECT')` (and INSERT/UPDATE), OR `aclexplode` the relation + column ACLs — so any object an app role can reach via ANY grant (table or column) enters the audit and must prove FORCE-RLS/allowlist. A test: a column-only `GRANT SELECT (secret) … TO kovo_reader`/`TO PUBLIC` over a non-RLS/owner table makes `kovo db check` REFUSE.

- [ ] **B2 — The KV435 secret-column audit matches a LITERAL `grantee = kovo_reader`, so `GRANT SELECT (secret_col) TO PUBLIC` (or to `kovo_writer`, or any role `kovo_reader` is a member of) re-exposes a redacted secret column to the reader on a GREEN audit.** (HIGH, framework/security, architectural; `pg-closure-audit-round2` PGCA-2; reproduced + root self-verified)
  - Observed: on a FORCE-RLS owner table `notes(id, owner_id, body, api_key)` (api_key is `secret`), `GRANT SELECT (api_key) ON notes TO PUBLIC`; the verbatim KV435 query (`postgres-runtime.ts:735-737`) returns 0 rows → `KV435 flags? false`. `information_schema.column_privileges` lists the api_key grantees as `['PUBLIC','postgres','kovo_writer']` — never literally `kovo_reader`. Then `SET ROLE kovo_reader; SET kovo.principal='alice'; SELECT owner_id, api_key FROM notes` returns `[{owner_id:'alice', api_key:'SECRET_ALICE'}]` — the redacted secret exposed.
  - Root cause (self-verified): `postgres-runtime.ts:731-746` verifies "kovo_reader must never hold SELECT on a secret column" via `information_schema.column_privileges WHERE … grantee = $readerRole` (`:735-737`). Postgres records a `TO PUBLIC` grant under `grantee='PUBLIC'` (and writer/other-role grants under their own names), which `kovo_reader` inherits — so the LITERAL grantee filter misses every indirect path. Same root as B1: the audit checks a literal predicate instead of EFFECTIVE privilege.
  - Why it matters: the framework's OWN column-REVOKE secret-column protection (the confidentiality mechanism secrets rely on) is defeated by a single `TO PUBLIC`/`TO writer` column grant — the reader effectively reads the secret while the audit stays green. Column-REVOKE (followup-6) is only as sound as the check that it stays revoked, and that check has the literal-grantee hole.
  - Repro evidence: `postgres-runtime.ts:735-737`; the column-only PUBLIC-grant read reproduced in `scratchpad/col-priv-probe.mjs` (reader reads a `GRANT SELECT (secret) … TO PUBLIC` column).
  - Acceptance: KV435 uses `has_column_privilege($readerRole, table, column, 'SELECT')` (effective privilege, resolves PUBLIC + membership) rather than a literal `grantee` match; a `GRANT SELECT (secret) … TO PUBLIC` makes the audit REFUSE. Shares B1's engine-oracle-at-column-granularity fix.

- [ ] **B3 — A cross-schema SECURITY DEFINER function escapes the routine closure audit (`WHERE nspname IN (app schemas)`), so a definer function in another schema that the reader can EXECUTE exfiltrates cross-owner data on a GREEN audit.** (HIGH, framework/security, architectural; `pg-provision-migrate-security` PMS-1; reproduced + root self-verified)
  - Observed: a `SECURITY DEFINER` function created in a non-app schema (owned by a superuser/owner) that `SELECT`s an owner table, with `EXECUTE` reachable by `kovo_reader`, runs as its definer (bypassing RLS) and returns cross-owner rows; `checkPostgresAppDbPosture` stays green because the routine scan never enumerates it.
  - Root cause (self-verified): `postgres-runtime.ts:989` — the `pg_proc` `prosecdef=true` enumeration is `WHERE n.nspname IN (${schemaPlaceholders})` (app schemas only), whereas the RELATION scan (`:799`) uses all non-system schemas (`nspname NOT IN ('pg_catalog','information_schema')`). The inconsistency means a definer function in a helper/other schema the app role can EXECUTE is invisible to the audit. followup-8 correctly ADDED definer-function auditing (a good fix for the round-11 class) but scoped it too narrowly.
  - Why it matters: SECURITY DEFINER functions are the canonical RLS bypass; auditing them was followup-8's own hardening, and the schema restriction reopens the exact class it closed, one schema over. Cross-schema helpers are a normal Postgres pattern.
  - Repro evidence: `postgres-runtime.ts:989` (routine scan app-schema-only) vs `:799` (relation scan all-schema).
  - Acceptance: the routine audit enumerates `pg_proc` across ALL non-system schemas (match the relation scan), testing `has_function_privilege(role, oid, 'EXECUTE')` and `prosecdef`, and refuses any reader/writer-EXECUTE-reachable definer routine not vetted. A test: a cross-schema definer function reachable by `kovo_reader` makes `kovo db check` REFUSE.

- [ ] **B4 — USAGE-only sequences escape the closure audit: a sequence (relkind `S`) granted `USAGE`/`SELECT` shows `has_table_privilege` all-false, so it is dropped from the reachable set and `kovo_reader` can read a sensitive counter's `currval`/`nextval`.** (LOW, framework/security; `pg-closure-audit-round2` PGCA-3; reproduced)
  - Observed: a sequence granted `USAGE` to `kovo_reader` is not enumerated by the `has_table_privilege` scan (sequences use `has_sequence_privilege`), so the reader reads `currval`/`nextval` of a counter that may reveal row counts / sequential IDs across tenants.
  - Root cause: same root as B1 — `has_table_privilege` does not cover sequences; the audit has no `has_sequence_privilege` path. (Note the DUAL over-block: legitimate serial/identity sequences the writer NEEDS are separately mis-flagged `KV433_REACHABLE_OBJECT 'unsupported relkind S'` → `papercuts-32` P2. Sequences are handled both too loosely for read and too strictly for writes.)
  - Why it matters: low severity (sequence metadata, not row data), but it is a reachable object class the audit's "deny anything not proven safe" contract silently skips.
  - Acceptance: sequences reachable by an app role are audited via `has_sequence_privilege` and either allowlisted (framework-owned serial sequences a writer needs) or refused; resolves both B4 (under-audit) and P2 (over-block) coherently.

- [ ] **B5 — The `NonRequestPrincipalPosture` brand check is not applied at the act-as principal-derivation consumer: a plain, UNBRANDED `principalPosture` attached to a DB request overrides the session principal, so any code path that can attach a posture impersonates any principal (reads any user's session/owner data).** (MED, framework/security; `pg-auth-session-security` PGAUTH-1; reproduced, verifier-corrected HIGH→MED for reachability)
  - Observed: `assertNonRequestPrincipalPosture` (`auth-principal.ts:148-157`) correctly requires the module-private WeakSet brand (`nonRequestPrincipalPostures.has(value)`), and `ctx.actAs(id)` mints a branded posture (`endpoint.ts:285`, attached at `:307`). But the principal-derivation path that reads `request.principalPosture` and honors `posture.kind === 'act-as' → posture.principal` does NOT call `assertNonRequestPrincipalPosture` first, so a hand-attached `{kind:'act-as', principal:'victim', audit:{…}}` is honored and the DB runs under `kovo.principal='victim'` — reading the victim's session token / owner rows.
  - Root cause (self-verified): the brand-assert seam exists (`auth-principal.ts:148`) but is not the sole door to principal derivation — the act-as consumer derives the principal from the posture's structural shape without the brand assertion. The brand (a module-private `unique symbol`/WeakSet) is exactly the CLAUDE.md unforgeable-capability pattern, and one consumer bypasses it.
  - Why it matters: the brand exists precisely to make a forged posture impossible; a consumer that skips it reintroduces principal impersonation. Reachability caveat (why MED not HIGH): `principalPosture` is attached server-side via `Object.defineProperty`, not from HTTP input, so it is app-code-reachable (a task/webhook/endpoint handler constructing a posture), not directly remote-attacker-reachable — analogous to the PGlite app-author-reachable class. Still a defense-in-depth hole in the auth principal seam.
  - Repro evidence: `auth-principal.ts:148-157` (brand assert) + `endpoint.ts:285-310` (mint+attach); the consumer honors an unbranded posture (verifier reproduced a plain posture overriding the session principal and reading another user's session token).
  - Acceptance: every consumer that derives a DB principal from a request/non-request posture calls `assertNonRequestPrincipalPosture` first (make the brand assert the sole door to principal elevation); a test asserts a hand-constructed unbranded posture is REJECTED, not honored.

## Refuted / Not Carried Forward (strong positive signal)
