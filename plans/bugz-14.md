# Bugz 14

Created 2026-06-29. Source of truth remains `SPEC.md`. Four confirmed security/soundness
defects escalated from the postgres-template dogfood sweep (companion papercuts ledger:
`plans/papercuts-super-5.md`). All are fail-OPEN/silent gaps in _normative blocking gates_
or the marquee update-coverage theorem, reproduced first-hand on fresh `create-kovo` (default
postgres/PGlite) apps. Two were self-verified by the main thread (B1 via static read-set +
drizzle export check; B2 via a live `vp dev` Playwright drive on a clean scaffold).

## Scope

- Apps: fresh `create-kovo` default (postgres/PGlite) scaffolds, link-local to the monorepo,
  exercised through `vp dev`, `kovo build`/`build:prod`, and authenticated HTTP/Playwright drives.
- Out of scope: the DX/diagnostic/read-shape papercuts (in `papercuts-super-5.md`).

## Issues

- [x] **B1 — RQB `with`-relation reads bypass the KV414 owner-scope IDOR gate on the pinned `drizzle-orm@1.0.0-rc.3` (cross-tenant read). Reopens `bugz-4` H2.** (HIGH, framework; found by `t3-relational`)
  - Exploit / observed: two byte-equivalent reads of an `owner:`-annotated `notes` child table behave asymmetrically. The `leftJoin` form is correctly gated — `build:prod` emits `KV414 QUERY … domain=model/note … (IDOR)` and `KV407` invalidation. The relational form `appDb.query.contacts.findMany({ with: { notes: true } })` gets **NO KV414 and NO KV407**: its static read set is `["model/contact"]` with `model/note` silently dropped, so an owner-scoped related table is never audited and any viewer can read every principal's `notes` rows.
  - Root cause: `packages/drizzle/src/static/schema.ts:188` (and the column-shape twin `:1406`) gate relation resolution on `relationCallName(call) !== 'relations'`, but the pinned `drizzle-orm@1.0.0-rc.3` does **not export `relations` at all** — `defineRelations` is the only relations API (`require('drizzle-orm')` → `relations:false, defineRelations:true`), and `defineRelations` appears nowhere in `packages/drizzle/src`. So the relation→table resolver map is always empty. The read-set fold `relationalReadWithObjectTableExpressions` (`project-receivers.ts:896`) and its twin (`summaries.ts:1379`) do `const table = relationTargetTableNames.get(relation); if (table) tables.push(table)` with **no `?? relation` fallback** (unlike the shape path `query-shapes.ts:191`), so even when the relation name equals the table name the empty resolver drops it. `scopeAuditsFromQueryFacts` only audits owner domains present in `reads` → the `with` table is never KV414-checked.
  - Why it matters: this is the identical exploit `bugz-4` H2 (`[x]` FIXED) — an owner-scoped table reached via RQB `with:` bypasses the IDOR gate and returns every principal's rows. The fix folded `with`-relation tables into the read set, but only for the obsolete `relations()` API; on the drizzle version the starter actually pins (where `defineRelations` is the _only_ option), the gate is fail-open again. SPEC §10.6 ("the JOIN is the declaration") makes the forgotten dependency representable and silent.
  - Repro evidence (self-verified): in `t3-relational` (schema uses `defineRelations({contacts,notes},…)`, the only available API) — `grep -ho '"query":"queries/contact[^}]*"reads":\[[^]]*\]' .kovo/cache/static-build-analysis/*.json` → `contacts-with-notes-query reads=["model/contact"]` (note dropped) vs `contact-notes-join-query reads=["model/contact","model/note"]`; `build:prod` emits KV414/KV407 for the join form only. `node -e "const d=require('drizzle-orm'); console.log('relations' in d, 'defineRelations' in d)"` → `false true`. Resolver gate confirmed at `schema.ts:188,:1406`.
  - Fix / acceptance: teach the relation-name resolver and read-set fold to understand `defineRelations()` (and add the `?? relation` fallback in `project-receivers.ts:896` / `summaries.ts:1379` so a name==table relation still folds). Prove with a build test: an owner-scoped `db.query.<t>.findMany({ with })` over a `defineRelations` schema folds the related owner table into `reads` and fires KV414, matching the `leftJoin` form.
  - Fixed evidence: `pnpm exec vitest run packages/drizzle/src/index.recognizer-alias-bugz3.test.ts --reporter=dot` proves a `defineRelations({ posts, comments }, …)` RQB `with` read folds `comment` into `reads` and emits the owner-scope audit input.

- [x] **B2 — The default starter's enhanced mutation SUCCESS path ships an unconsumable `<kovo-query>` delta instead of `<kovo-fragment>` HTML, so a committed row is invisible until full reload — a silent stale-UI violation of the §10.6/KV311 update-coverage theorem.** (HIGH, framework; found by `t4-optimistic`)
  - Exploit / observed: on the **unmodified default scaffold**, signed in, submit the shipped enhanced "add contact" form. The client correctly sends `Kovo-Fragment: true` + `Kovo-Targets: contacts-region=queries/contacts-query` + `Kovo-Live-Targets` + `Accept: text/vnd.kovo.fragment+html`. The server commits the row (response carries the new row; `Kovo-Changes: [{domain:model/contact}]`) but the body is **only** `<kovo-query name="queries/contacts-query">{…}</kovo-query>` with **no `<kovo-fragment>`** (despite `Content-Type: text/vnd.kovo.fragment+html`). The fragment-status region has no client query store and no `data-bind` stamp, so the query delta has nothing to apply to → the DOM never updates. Decisive contrast: a DUPLICATE-email submit (422) DOES emit a full `<kovo-fragment target="contacts-region">…</kovo-fragment>` and the region morphs — so the live-target renderer exists and works; only the SUCCESS path fails to invoke it.
  - Root cause: the live-target descriptor for `contacts-region` carries `descriptorRerunsSizes=[0]` → `OUT_liveTargetDescriptors=[]` (fragment suppressed), while `OUT_rerunQueries=[{key:queries/contacts-query}]` (a query delta is emitted via the plan-covered branch because no fragment renderer matched). Proximate cause: the live-target renderer's `queryBindings[0].query.reads` is **empty `[]`**, whereas the registry `queryDefinition` has the compiler-extracted `reads=[{key:model/contact}]`; so `liveTargetDescriptorQueryReruns → queryTouchedByChange` never matches `change.domain=model/contact` → empty reruns → the descriptor fragment is dropped and only the unconsumable query delta is sent.
  - Why it matters: this is the framework's headline differentiator (SPEC §10.6 / KV311: every query/state-dependent DOM position has a declared update status or the page never renders data it can't refresh; SPEC §8 line 442: fragment status = mutation success re-renders the target). The DEFAULT starter's core "add a contact and watch it appear" flow is broken — every `create-kovo` user hits an invisible-until-reload write on first use. Likely a regression (prior `papercut-super-codex-1.md:61` recorded fragment reconcile working).
  - Repro evidence (self-verified, clean base scaffold): `vp dev` on a fresh `create-kovo` app, `KOVO_DEMO_PASSWORD` set, login `demo@example.com`, submit the enhanced add-contact form via Playwright → request headers as above; **response body = `<kovo-query>` only, no `<kovo-fragment>`**; new row present in the body but **final DOM = 3 rows / "3 contacts"** (the committed 4th row never appears). Error path (422) emits a fragment and morphs.
  - Fix / acceptance: the success path of an enhanced mutation targeting a fragment-status region must emit the `<kovo-fragment>` re-render (the same path the 422 error case uses), or the live-target binding's `reads` must be populated so `queryTouchedByChange` matches the change domain. Prove with a runtime/e2e test: a successful add-contact updates the `contacts-region` DOM with no reload.
  - Fixed evidence: `pnpm exec vitest run packages/server/src/live-target-renderer.test.tsx packages/server/src/mutation-response.test.ts --reporter=dot` proves compiler-emitted live-target query bindings fold generated reads and the enhanced success response includes a matching `<kovo-fragment>`.

- [x] **B3 — The KV302 binding-shape gate validates only the FIRST component in a multi-component module; 2nd+ region bindings are silently unvalidated (fail-open soundness gate).** (MED, framework; found by `t5-multiregion`)
  - Exploit / observed: a module defining multiple `component()` exports (a multi-region page) only has its first-defined component's `data-bind` paths validated. With two regions `RegionA`/`RegionB` both binding the same query, adding an out-of-shape field `extra` (in the TS type but not the query `output` schema) and binding `{a.extra}` + `{b.extra}`, `build:prod` flags `a.extra` but **does not flag `b.extra`** — the out-of-shape path in the second region passes the build with no diagnostic. Swapping definition order moves the failure to whichever component is first.
  - Root cause: `packages/compiler/src/scan/parse.ts:299-300` (`firstComponentModel` returns `model.components[0]`); `componentOptionObjectKeys`/`componentQueryNames` (`parse.ts:323-340`, `analyze/query-shapes.ts:42-44`) and the alias resolution in `componentLocalQueryShapeFacts` (`vite.ts:692`) read only that first component. So only `component[0]`'s `queries` aliases become known query shapes; bindings in 2nd+ components reference aliases that are never registered and never proven → no KV302. Lowering and the other validators DO iterate all `model.components` (`compile.ts:747/762`, `app-graph.ts:529/672`), so multi-component modules are first-class — the gap is specific to the query-shape/binding-validation path.
  - Why it matters: KV302 is part of the SPEC §5.2 soundness proof that server render and client update read the _same declared shape_. On any multi-region page (the canonical §9.5 request-shell composition), all but the first region ship binding paths the gate never proved — re-introducing the exact class of stale/undefined-bind bugs the gate exists to prevent.
  - Repro evidence: `t5-multiregion/src/components/dashboard.tsx` with two `component()` exports both binding `metaQuery` (whose `output` omits `extra`), `rm -rf .kovo/cache node_modules/.vite dist && pnpm run build:prod` → KV302 only for `component[0]`'s aliases, nothing for the second region; order swap flips it. Root cause source-confirmed at `parse.ts:300`.
  - Fix / acceptance: the query-shape/binding validator must iterate every `model.components[i]` (not just `[0]`). Prove with a build test: an out-of-shape binding in the 2nd component of a multi-component module fails KV302.
  - Fixed evidence: `pnpm exec vitest run packages/compiler/src/query-bindings.test.ts packages/compiler/src/vite.test.ts --reporter=dot` proves Vite aliases every component's query shapes and an out-of-shape second-component binding reports KV302.

- [x] **B4 — The compiler's `kovo-props` host stamp emits declared prop names as bare shorthand identifiers, so any query-bound component whose render doesn't destructure EVERY prop ships a guaranteed server-side `ReferenceError` 500 — with a green `tsc` and a green `kovo build`.** (MED, framework; found by `t1-pagination`)
  - Exploit / observed: a `component({ props, queries, render })` whose render destructures only `{ contacts }` (the idiomatic shape when props feed query args, not the render body) renders a 500 at runtime: `ReferenceError: page is not defined` at the render call. The lowered module stamps the component-root element `kovo-props={JSON.stringify({ page, pageSize, sort, dir, q })}` — bare identifiers that execute in the render scope. `tsc --noEmit` is green and `kovo build ./src/app.tsx` exits 0 with the broken render, so neither the type check nor the deploy gate catches it.
  - Root cause: `packages/compiler/src/emit/server-render.ts:459-466` (`componentPropsStamp`) builds `const props = componentOptionObjectKeys(model, 'props')` and emits `value: \`JSON.stringify({ ${props.join(', ')} })\``(valueKind`'expression'`) unconditionally for every declared prop whenever the component is a server-refresh target — it never checks those names are bound render params. But `render({ ...props, ...queries }, …)` (`packages/server/src/jsx-runtime.ts:626`) puts prop *values* inside arg0, reachable only if the author destructures them. The coverage validator KV303 (`packages/compiler/src/validate/component-contracts.ts:154-227`) only flags names the render READS or RENAMES; it does **not** enforce that declared props injected into the stamp are bound render params, so the omission case injects unbound identifiers with zero diagnostic (while the related rename case IS diagnosed) — a fail-open coverage gap.
  - Why it matters: a latent production 500 that both the deploy gate and `tsc` certify green. Declaring `props` for fragment/live serialization while reading them only as query args (the normal pagination/search-param authoring path) sits directly on this trap; the author sees a blank "Server Error" with no compile error and (with B-pg5-t1-3) an empty terminal.
  - Repro evidence: `t1-pagination` `BrowseContacts` declares `props: { page, … }` + `queries: { contacts: pagedContactsQuery.args((p)=>({page:p.page,…})) }`, render destructures `{ contacts }` only → `tsc` green, `build:prod` exit 0, `GET /browse` (authed) 500s with `ReferenceError: page is not defined`. Emitter + runtime-merge source confirmed at `server-render.ts:459-466` and `jsx-runtime.ts:626`; the compiler cache holds the lowered `kovo-props={JSON.stringify({ page, pageSize, sort, dir, q })}` against a `render({ contacts })` signature.
  - Fix / acceptance: the stamp must reference the actual first-arg props object (e.g. read from `arg0`/a stable props binding) rather than bare shorthand identifiers, OR KV303 must fail the build when a declared prop injected into the stamp is not a bound render param. Prove with a compile test: a component declaring props it does not destructure either lowers to a valid stamp or emits a diagnostic — never an unbound identifier.
  - Fixed evidence: `pnpm exec vitest run packages/compiler/src/stamps.test.ts --reporter=dot` proves compiler output no longer emits a `kovo-props={JSON.stringify({ page… })}` shorthand stamp for query-only props; runtime root stamping remains covered by server tests.
