# Round-5 Soundness Bugz 26

Created 2026-07-01. Source of truth remains `SPEC.md`. Security/soundness defects found dogfooding AFTER
`plans/fundamental-fixes-followup.md` was implemented (census + statement-parse guards + resolver table +
value-flow-fail-closed + mutation testing). Papercuts in `plans/claude-papercuts-24.md`.

**Meta-theme — the followup's guards HOLD for what they enumerate, but the same enumerate-and-allow pattern
persists one layer deeper.** The enumerated surfaces largely held (see the long Refuted list: read-only handle
DML, `.all/.get/.transaction/$client`, method-extraction, relational-query-builder, KV426 alias/cast/comma,
CTE/`INSERT…SELECT` row-level, `/_q` cache headers, redirect/Set-Cookie/CRLF, egress floor, auth, enhanced
success body, CSRF, cache floor). The four holes below are on the axes the guards do NOT fully enumerate:
statement **TYPE** (the parser covers only DML → DDL bypasses), expression **FORM** (object-literal member,
arrow-closure), and gate **SCOPE** (KV435 is query-only, not mutation handlers). The census's own completeness
gate doesn't enforce a required-set for the two hand-enumerated kinds, so it certified `complete:true` while
these gaps existed (`claude-papercuts-24` P1). Root causes confirmed first-hand; B1 self-verified on both
enforcement sites; B2–B4 reproduced by independent verifiers via isolation flips.

## Issues

- [ ] **B1 — DDL (`DROP`/`ALTER`/`CREATE TABLE`) bypasses the primary statement-parse guard: the read-only floor (KV433) AND the declared-tables allowlist (KV406) both fail OPEN. A read-only handle can DROP any table; a mutation can DROP/ALTER a table outside its declared set.** (HIGH, framework/security; found INDEPENDENTLY by `statement-parser-deep` + `read-handle-deep`, SELF-VERIFIED both sites)
  - Observed: `readonlyDb(db).run(sql`DELETE FROM contacts`)` throws (KV433, the H fix holds), but `.run(sql`DROP TABLE contacts`)` **executes** (table count 1→0). A managed write handle with `tables:['contacts']` throws on `UPDATE userx` (KV406) but **executes `DROP TABLE userx`** (undeclared table dropped, count 2→1). Both branded Kovo `sql`/`trustedSql` (pass the KV422 brand check), then bypass the table check.
  - Root cause: `packages/server/src/sql-write-allowlist.ts:119-134` `writeTablesForStatement` switches only on `insert/update/delete/truncate table/with/with recursive`; `drop table`/`alter table`/`create table` (which `pgsql-ast-parser` parses fine) fall to `default: return []`. Both enforcement sites in `packages/server/src/sql-safe-handle.ts` treat `writeTables.length === 0` as not-a-write — `assertReadSqlStatement` (`:582`, the read-only floor) and `assertSqlWriteTablesAllowed` (`:550`, the declared-tables allowlist) — so a branded DDL statement is allowed on both the `Reader<Db>` floor and the mutation write handle.
  - Why it matters: `readonlyAppDb` (a query loader's `context.db`, and the blessed endpoint-read handle) can `DROP`/`ALTER`/`CREATE` any table — schema destruction / a `CREATE TABLE` staging a subsequent write — on a green build. A mutation's declared `tables:` allowlist (the §11.2 `observed ⊆ declared` contract) is void for DDL: a mutation scoped to `contacts` can `DROP TABLE user` or `ALTER TABLE account ADD COLUMN backdoor`. The parser is the *primary* guard (H/I), so its DML-only coverage is a direct fail-open.
  - Repro evidence (SELF-VERIFIED): read-only floor — `.run(ksql`DELETE`)` throws, `.run(ksql`DROP TABLE contacts`)` → count 1→0; managed write handle `tables:['contacts']` — `UPDATE userx` throws KV406, `DROP TABLE userx` → count 2→1. Parser: `parseSqlWriteTables('DROP TABLE contacts',{dialect:'sqlite'}) === []` vs `'DELETE FROM contacts' === ['contacts']`.
  - Acceptance: `writeTablesForStatement` classifies DDL (`drop/alter/create table`, and `create/drop index/trigger/view`, `attach`, `pragma`-write) as writes (or the enforcement sites fail closed on any statement type the classifier does not positively prove read-only), so DDL on a read handle fails KV433 and DDL outside `tables:` fails KV406. Add a per-statement-type metamorphic seed (DDL × dialect).

- [ ] **B2 — KV426 trusted-HTML XSS gate fails OPEN on an object-literal member table (`const trust = { html: trustedHtml }; trust.html(taint)`): an unrecognized property-access callee is SKIPPED, not failed closed, so reflected XSS ships on a green build.** (HIGH, framework/security; found by `resolver-correctness-deep`, verified independently)
  - Observed: direct `trustedHtml(request.headers.get('x-xss') ?? '')` → KV426 RED; `const trust = { html: trustedHtml }; trust.html(reflected)` → GREEN, and the prod artifact reflects `<script>alert(document.cookie)</script>` from an `x-xss` header verbatim into the SSR document (raw `<script>` count 1). Identifier alias / `as`-cast / comma-operator callees are all correctly caught — only the object-literal member escapes.
  - Root cause: the KV426 sink recognizer `rawTrustSinkForCall`/`rawTrustSinkForExpression` (`packages/compiler/src/validate/trusted-html-provenance.ts:119-142`) recognizes a trust sink only via (a) the brand resolver, (b) dynamic namespace element-access, or (c) a same-file identifier wrapper. A property-access callee on a local object matches none, returns null, and the call is **skipped** (`:56-72`) — there is no "unrecognized callable that could carry a trust brand ⇒ un-provable ⇒ fail-closed" branch (violating the followup's own fail-closed-on-unprovable principle). Underlying resolver gap: `packages/core/src/internal/framework-identity.ts:452-503` `namespaceMemberIdentity` does not resolve a property-access whose receiver is a variable initialized to an object literal binding the brand.
  - Why it matters: the KV426 stored/reflected-XSS gate is bypassable by assigning `trustedHtml` to an object property — a natural "render table" pattern — shipping an unsanitized raw-HTML sink for request-derived data on a green build.
  - Repro evidence: build flip — object-literal member GREEN vs direct RED; prod artifact reflects the `x-xss` header `<script>` unescaped.
  - Acceptance: the KV426 recognizer resolves an object-literal member to the brand (via the resolver) AND fails closed on any *unrecognized* callee that could carry a trust brand (call to a value the resolver cannot prove non-trust ⇒ un-provable ⇒ KV426), rather than silently skipping. Metamorphic seed: object-literal member + arbitrary unrecognized callee.

- [ ] **B3 — KV435 secret-to-wire misses an arrow-closure-in-const invoked with no args: a secret column captured by closure and written to the returned wire shape ships on a green build.** (HIGH, framework/security; found by `value-flow-deep`, verified independently; incomplete-fix residual of the KV435 cross-select fix)
  - Observed: a loader with a clean wire-shape select plus a second `{id, secret: contacts.apiKey}` select whose rows are `push`ed into the returned array. Top-level loop → KV435 RED. Wrapping the identical loop in `const collect = () => { for (const r of secretRows) out.push({name: r.secret, …}); }; collect();` → GREEN, and the prod artifact serves the secret bytes over `/_q`.
  - Root cause: `packages/drizzle/src/static/query-shapes.ts` off-wire taint tracker (`taintedValueReachesWire`) recurses into nested function bodies only for named `FunctionDeclaration`s (`:419-435`), and gates local-function-body analysis behind `if (!taintedArgs) continue` (`:481`). An arrow assigned to a const, invoked with **no arguments**, that captures the tainted select result by closure and writes it into a wire-root accumulator, enters neither path; `topLevelDescendant` (`:774`) also excludes the nested write. `taintedValueReachesWire` returns false → the secret projection is treated as off-wire.
  - Why it matters: the KV435 value-flow gate the followup claims is "fail-closed on un-provable, taint through every form" is defeated by wrapping the wire-write in a zero-arg arrow closure — a trivial refactor.
  - Repro evidence: build flip — arrow-closure GREEN vs top-level loop KV435 RED; prod artifact `curl /leak` returns `sk_live_…` secret values.
  - Acceptance: the taint tracker recurses into arrow/function-expression bodies (invoked with any or no args) and treats a closure-captured tainted value written to a wire-root accumulator as reaching the wire; an un-analyzable closure fails closed (KV406). Metamorphic seed: closure/generator/promise capture forms.

- [ ] **B4 — KV435 secret-to-wire does NOT cover mutation handlers: a mutation reading a secret column and returning it (routed to the wire via `redirectTo(result)` / `stream({result})`) ships the secret on a green build.** (MED, framework/security; found by `uncovered-surfaces-deep`, verified independently; new)
  - Observed: a query loader projecting `session.token` → KV435 RED. The identical read in a `mutation({ handler })` returning `{ tokens: rows.map(r => r.token) }` builds fully green (`pnpm run check` PASS). The default mutation body doesn't echo the return, but `redirectTo:(result)=>…` (`mutation.ts:241,305`) and `stream({result})` (`mutation.ts:884`) route `MutationSuccess.value` to the wire.
  - Root cause: `packages/compiler/src/validate/confidentiality.ts:9` `validateSecretQueryWire` operates only on extracted query shapes; the secret value-flow walk (`query-shapes.ts:2540 secretValueFlowProjectedColumns`) is invoked from query-loader shape extraction only. Mutation handlers get read/touch extraction for invalidation but no secret→result→wire taint, and there is no gate for `MutationSuccess.value` flowing into `redirectTo`/`stream`.
  - Why it matters: a whole wire channel (mutation response value via `redirectTo`/`stream`) is exempt from the secret-to-wire gate — the same KV435 leak the followup closed for queries is open for mutations.
  - Repro evidence: `pnpm run check` PASS on the mutation-handler secret read; the identical select in a query loader → KV435 at `build:prod`.
  - Acceptance: KV435 secret value-flow covers mutation handler return values that reach the wire via `redirectTo`/`stream`; a mutation returning a secret column fails closed. (Belongs in the census `output-wire-sink` family — see `claude-papercuts-24` P1.)
