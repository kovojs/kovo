## Background — why this is a workstream

The framework has no public-facing explanation of itself: `README.md` points at `SPEC.md`, which is normative, dense, and written for implementers. Adoption (and the P10 outside-legibility study) needs a site that explains jiso to humans and agents who haven't read the spec. Equally important, D8 R7 names a docs site as the static exporter's first outside consumer — the site is the proof that `vp run export` produces something real. The two needs are one deliverable: **the docs site is a jiso app**, exported statically, L0/L1 only by D8's stated constraint, and itself evidence for every claim it makes (view-source is readable, it works with JS disabled, the loader is 4KB, TTI ≡ FCP).

The tagline is fixed: **"The TypeScript web framework where agents get build-time errors and users get instant pages."** The landing page explains it in three sections — _Agents get build-time errors_, _Users get instant pages_, _Developers get great debugging experience_ — and the docs proper are four tracks: Getting Started, Tutorial, Guides, API reference.

### Decisions (recorded so we don't relitigate)

- **Visual direction: polished product.** TanStack/Astro-tier design on the starter palette (`--color-jiso-ink: #17202a`, `--color-jiso-accent: #0f8b8d`), real visual hierarchy — but **every visual is a real artifact** (styled terminal captures, Network-panel traces, `fw explain` output). No abstract illustrations. (Rejected: text-first terse — undersells to the evaluation visit; spec-grade minimal — right for `/spec`, cold for a landing page.)
- **Proof is captured, not written.** The three landing sections show genuine toolchain output regenerated at every site build: a deliberately broken fixture run through `vp check` for the FW teaching error; the pinned wire fixtures for the enhanced-mutation Network trace; the P10 perf-gate numbers for the instant-pages claims. Drift-proof by construction — if a capture source disappears or its output changes shape, the site build fails, not the claim. (Rejected: live interactive demos — exported sites are L0/L1, so mutation demos would need a second, server-backed deployment; revisit post-launch. Rejected: hand-written snippets — claims without evidence, the worst fit for this framework's pitch.)
- **Every tutorial step compiles and tests.** The tutorial builds the commerce app (the SPEC §16 acceptance target) step by step; each chapter is a checked-in code state that passes `vp check` and its own tests in CI, and the final state behavior-matches `examples/commerce`. Prose snippets are extracted from step states, never hand-written. Same doctrine as the wire fixtures: the tutorial cannot drift, it can only fail loudly. Pre-1.0 API churn makes this the only tutorial worth writing.
- **API reference scope: the five app-facing packages**, generated from TypeScript declarations + doc comments so it cannot drift from source. Undocumented public exports render loudly as undocumented rather than being omitted. **Not at launch** (offered, not chosen): generated FW-diagnostics reference pages and the `fw` CLI / graph-format reference — both remain natural later additions since their sources (the diagnostics registry, the `fw-explain/v1` snapshots) are already machine-readable.
- **Agent surface: `llms.txt` + markdown mirrors.** Every docs page resolves as raw markdown at a stable URL (`/docs/<page>.md`); `llms.txt` indexes the site. Cheap — the source _is_ markdown — and makes the tagline's first clause concrete. (Deferred: a "docs for agents" section with `fw explain` driving recipes and CLAUDE.md-ready snippets; add on real agent traffic.)
- **Search: build-time index + L1 island.** Pagefind-style index emitted at export; the search UI is an L1 island loaded on first ⌘K — zero JS before interaction, zero server, and a genuine showcase of the island model on the site itself. (Rejected: hosted search — a third-party script the legibility story would have to apologize for.)
- **`/spec` publishes `SPEC.md` verbatim**, rendered at build time from the repo file with stable § anchors. Docs pages cite §-anchors as links; the docs explain, the spec stays the single normative source — the same hierarchy `CLAUDE.md` imposes on this repo. P10's docs freeze applies to the spec content, not to this rendering path.
- **The markdown pipeline is site-local build tooling, not framework surface.** A markdown library + shiki run inside the site's build; the framework gains no markdown feature (SPEC §4.1's position holds: behavior attributes work on markup jiso didn't render). If a second jiso site wants the pipeline later, extraction into a package is a refactor, not a design change.
- **IA is fixed at four tracks.** Getting Started (install, starter walkthrough, the mental model: components → IR → self-describing HTML); Tutorial (commerce, chapters roughly: scaffold → first page → islands → queries → mutations & forms → invalidation → optimism → streaming → testing → deploy); Guides (the eight above — task-oriented, each anchored to `fw explain` output where the graph is the explanation); API reference (generated).

## W-track

- **W1 — Scaffold.** `site/` workspace package consuming `workspace:*` jiso packages; route table for the five sections; document override providing docs chrome (header, sidebar slot, footer) on top of the shell's assembled parts; Tailwind theme extending the starter tokens. Ships rendering placeholder content the moment D8 R5 (dev middleware) exists; export waits on R6.
- **W2 — Content pipeline.** Build step: markdown files with frontmatter (title, section, order, description) → lowered jiso page components; shiki at build time; § citations (`SPEC §6.4`) auto-link to `/spec#6-4`; sidebar and prev/next generated from frontmatter; every page also emitted as raw `.md` at a stable URL; `llms.txt` generated from the same index. Headings get stable anchor slugs (the link checker in W9 depends on it).
- **W3 — Artifact capture.** A `site/capture/` harness with one entry per artifact: each runs real tooling (a broken-on-purpose fixture through `vp check`; the P0 wire fixtures rendered as a Network-panel-styled trace; `fw explain mutation cart/add --optimistic` against the commerce graph; the P10 perf numbers) and emits HTML/text the pages embed. Captures regenerate on every build; a capture whose source can't produce output fails the build. Styled presentation, verbatim content.
- **W4 — Landing.** Hero (tagline, `Get started`, `pnpm create jiso` one-liner) + three sections: **Agents get build-time errors** — the FW227-class teaching error capture plus the rename-propagation story (rename a route, every `<Link>` goes red) and `fw explain` as the queryable app graph; **Users get instant pages** — MPA first paint with zero hydration, the 4KB loader budget, TTI ≡ FCP and prerendered-nav numbers from the perf gates, works-with-JS-disabled; **Developers get great debugging experience** — the readable wire (named POSTs, `FW-Idem`/`FW-Targets`/`FW-Changes` headers, fragment responses you can read in the Network panel), view-source as documentation, diffable `fw check` output in CI.
- **W5 — Tutorial.** Step states under `site/tutorial/steps/NN-name/`, each a minimal workspace member compiling under `vp check` with per-step tests; a snippet extractor pulls marked regions into the chapter markdown; CI job runs all steps; final step asserts behavior parity with `examples/commerce` (reusing its matrix tests where possible). Chapter prose explains _why_ at each step, citing SPEC §s.
- **W6 — API reference.** Generator reads the five packages' public `.d.ts` + doc comments and emits one markdown page per package (per-export entries: signature, doc comment, SPEC § citation when the comment carries one) into the W2 pipeline — so the reference automatically participates in search, `.md` mirrors, and `llms.txt`. Doc-comment gaps render as explicit "undocumented" entries; closing them is ordinary package work the reference makes visible.
- **W7 — Guides.** The eight launch guides. House rule: every guide that states a behavior cites the SPEC § and, where the behavior is graph-visible, embeds a W3-captured `fw explain`/`fw check` artifact instead of describing output.
- **W8 — Search.** Export-time index over the rendered pages; ⌘K/button opens an L1 island that lazy-loads the index; no JS before first use. The island's handler module is itself a nice `/c/` teaching example — link to its source from the Getting Started mental-model page.
- **W9 — Launch gates.** Clean-checkout `vp run export` builds the whole site; automated no-JS crawl (every page renders, every nav works); link + § anchor checker; `llms.txt`/mirror validation; Lighthouse pass recorded; published-site evidence feeds the P10 outside-legibility ledger.

## Out of scope

Live server-backed demos (exported site is L0/L1; a demo deployment is a post-launch follow-up) · FW-diagnostics and CLI/graph-format reference sections (deferred, sources already machine-readable) · versioned docs (pre-1.0, single version) · blog/changelog infrastructure · i18n of docs content · the "docs for agents" section beyond `llms.txt` + mirrors · domain/trademark/npm checks (`docs/prelaunch-checklist.md`).

## Sequencing & dependencies

- **Hard dependency: D8.** W1 needs R5 (dev middleware) to be workable and R6 (export) to ship; W9 needs R6/R7 complete. This plan should not start its W1 slice before the D8 S8 spike freezes `createApp()`.
- **Export-independent work can start now:** W2's markdown pipeline, W3's capture harness, W5's step states, and W6's generator are build scripts plus content — none need a serving framework. Content writing (W4 copy, W7 guides) is similarly unblocked.
- W2 precedes W6/W7 (they emit into it). W3 precedes W4. W8 lands after W2 has real content volume.
- The site joins root CI (`vp run acceptance` gains a site-build + tutorial-steps gate) only once D8 R6 lands, to avoid gating the framework on its own docs before the exporter exists.

## Exit criteria

1. Clean checkout → `vp run export` in `site/` → a static site that deploys to any static host and passes the automated no-JS crawl end-to-end.
2. Landing artifacts regenerate from the real toolchain at every build; removing a capture source (e.g. the broken fixture behind the FW teaching error) fails the site build rather than shipping a stale image.
3. Every tutorial step passes `vp check` and its tests in CI; an API change that breaks a step turns the tutorial red in the same PR; the final step behavior-matches `examples/commerce`.
4. The API reference is regenerated from declarations on every build; renaming a public export changes the reference in the same PR with no manual edit; undocumented exports are visibly flagged, not omitted.
5. `llms.txt` resolves and indexes the site; every docs page serves a raw `.md` mirror at a stable URL; the ⌘K island loads zero JS before first use.
6. `/spec` renders `SPEC.md` verbatim with stable § anchors and every docs § citation links into it.
7. The published site is sufficient input for the P10 outside-legibility study, and its evidence is recorded in the P10 ledger.
