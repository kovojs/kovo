# Bugz 17

Created 2026-06-29. Source of truth remains `SPEC.md`. Two confirmed security/soundness defects
escalated from the postgres-template dogfood sweep, both reproduced first-hand by the main thread
against a clean `create-kovo` (default postgres) **prod build**.

- **B1 is a fail-OPEN security defect** (cross-principal disclosure / shared-cache poisoning): the
  §9.4 public-read cache relaxation is gated on the author's non-enforcing `publicAccess()` assertion
  instead of the SPEC-mandated compiler session-independence proof.
- **B2 is a soundness REGRESSION**: `bugz-16` B1 was marked `[x]` fixed (unit tests) but still
  reproduces end-to-end in the prod artifact — the fix did not change the shipped behavior.

## Scope

- App: fresh `create-kovo` default (postgres/PGlite) scaffolds, link-local to the monorepo, built with
  `pnpm run build:prod` and run as the README prod command (`NODE_ENV=production node dist/server/server.mjs`).
- Out of scope: the island-deploy / caching-hygiene / static-export / streaming papercuts.

## Issues

- [ ] **B1 — The §9.4 public-read cache relaxation gates on the author's `publicAccess()` assertion, not the mandated compiler session-independence proof, so an unguarded query that reads `req.session` and sets `read.cacheControl: 'public, …'` is served `public, max-age` (no `Vary`) to shared caches — one principal's data is served to every other principal (cross-principal disclosure / cache poisoning).** (HIGH, framework; found by `t2-http-caching`)
  - Exploit / observed: a query with `access: publicAccess('…')` + `read: { cacheControl: 'public, max-age=300' }` + a `load` that returns `context.request.session?.user?.id` (no `guard`). Against the prod node build: **anon** `GET /_q/queries/leaky-mine-query` → `200`, `cache-control: public, max-age=300`, **no `Vary`**, body `{"userId":"anon"}`; **authed** same URL → `200`, identical `cache-control: public, max-age=300`, **no `Vary`**, body `{"userId":"QD3TyIjLEiPwQh6Kx0O7ex9UnRMu5g0n"}` (the real session user id). A shared CDN/proxy keys only on the URL, stores whichever response arrives first, and serves one principal's id to every other principal (and to anon) for 300 s. `pnpm run check` / `build:prod` pass with **zero** diagnostics. The genuinely public `publicStatsQuery` (no session read) correctly gets `public, max-age=60`, confirming the relaxation is reachable and the leak is the session-reading variant.
  - Root cause: `packages/server/src/query.ts:951-973` `querySuccessCacheHeaders` emits the public cache header gated **only** on `cacheControl && definition.guard === undefined && definition.access?.kind === 'public' && safeHeaderValue(cacheControl)` (called at `:690` with just `definition`); it never inspects `load` or the instance key for `req.session` reads. `access` is author metadata (`packages/server/src/access.ts:30-42`; `publicAccess(reason)` → `{kind:'public',reason}`, doc-commented "intentionally not an executable policy engine"). SPEC §9.4 (`SPEC.md:954`) requires the relaxation **only** for a query the **compiler proves** session-independent (no guard and no `req.session` read in key or load), mirroring §9.5/KV229 — but `grep -rn cacheControl packages/compiler/src` is empty (no such proof is emitted; `static-export-route-plan.ts:28`'s KV229 message itself says to "wait for compiler-backed session-dependence metadata"). So the runtime substitutes the non-enforcing `publicAccess` assertion for the mandated compiler proof, fail-open, with no diagnostic.
  - Why it matters: this is a fail-open downgrade of a security-critical default — the safe `private, no-store` posture flips to a shared-cacheable `public, max-age` response on author metadata alone, with no framework verification and no diagnostic. It defeats SPEC §9.4's closing guarantee ("a guarded query may never be served from a shared cache … must not be bypassable by an intermediary") via the adjacent unguarded-but-session-reading path. Realistic trigger: a legitimately public read with `read.cacheControl` later grows a personalization branch that reads session — nothing re-checks, and the next deploy silently leaks per-user data through any CDN/proxy. `CLAUDE.md`'s honesty boundary states asserted metadata is an author-time guardrail, **never** the security proof; here it is the only gate on a shared-cache decision.
  - Repro evidence (self-verified, clean base scaffold, prod): authored `leakyMineQuery` (above), `build:prod` exit 0 (no diagnostic), `NODE_ENV=production node dist/server/server.mjs`; anon vs authed `curl -i /_q/queries/leaky-mine-query` both return `cache-control: public, max-age=300` with no `Vary`, authed body leaks the session `userId`.
  - Fix / acceptance: gate the public-cache relaxation on a compiler proof of session-independence (no `req.session` read in key/load and no guard), not on `publicAccess` metadata — and emit a diagnostic (KV-class) when a query sets a shared-cache `cacheControl` while reading session/being guarded; at minimum force `private, no-store` (or add `Vary: Cookie`) for any session-reading query regardless of `access`. Prove with a build/runtime test: a session-reading query with `cacheControl: 'public'` either fails the build with a diagnostic or is served `private, no-store`.

- [ ] **B2 — `bugz-16` B1 regression: the production `kovo build` artifact STILL ships an empty enhanced-mutation success body, so the committed row never renders — the closed fix (unit tests) did not change the shipped behavior.** (HIGH, framework; regression of `bugz-16` B1; found in super-8 baseline)
  - Observed / impact: on the **unmodified default scaffold**, clean prod build, signed in, submitting the shipped enhanced add-contact form returns `200` with `Content-Type: text/vnd.kovo.fragment+html` and `Kovo-Changes: [{"domain":"model/contact"}]`, but the **response body is still empty** (no `<kovo-query>`/`<kovo-fragment>`), and the DOM stays at the pre-submit rows (`3 rows / "3 contacts"`). Identical to `bugz-16` B1, which was marked `[x]` fixed with a passing unit suite (`index.kovo-build.test.ts` / `vite-data-plane-gate.test.ts`).
  - Root cause: the `bugz-16` fix added unit coverage but the prod **artifact** behavior is unchanged — the registry-wrapped fragment region's live-target renderer is still not lowered/registered in the bundled handler, so `renderSuccessfulMutationWireResponse` (`packages/server/src/mutation.ts:551-614`) still joins empty `queryChunks + fragmentChunks` for `contacts-region` (the shared root cause with `papercuts-super-7` A1 / the prod-bundle reentry guard). The unit tests assert renderer/build internals, not the end-to-end prod-server success body that `bugz-16`'s own acceptance demanded ("prove with an end-to-end prod-artifact test, not a unit test of the renderer").
  - Why it matters: `build:prod` is the only deploy path and this is the canonical starter pattern (a server-refreshable fragment region); every deployed app's core "submit a mutation, see the result" flow silently fails, the §10.6/KV311 update-coverage theorem is violated in the shipped artifact, and the regression is invisible to the dev loop and the unit suite that marked it closed.
  - Repro evidence (self-verified, clean base scaffold, prod, after a fresh `rm -rf dist && pnpm run build:prod`): Playwright login + submit add-contact → success response body empty (`has <kovo-fragment>` = false), final DOM unchanged; the 422 path still emits a fragment and morphs.
  - Fix / acceptance: as `bugz-16` B1 — but the acceptance test **must** be an end-to-end prod-artifact assertion (build a starter, run `node dist/server/server.mjs`, submit a mutation over HTTP/Playwright, assert the success response carries a non-empty fragment and the DOM updates). Do not re-close on unit-test evidence.
