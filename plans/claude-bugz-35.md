**Meta-theme — followup-9's own C7 invariant ("no security decision on a necessary-but-insufficient condition")
reproduced on followup-9's OWN mechanisms.** followup-9 hardened the closure audit and REVOKEd `set_config` EXECUTE to
stop principal/role forgery. But the owner-scoping + crossOwnerRead model rests on two things the fix did not actually
confine, and the routine audit still uses a proxy:

| Finding | The necessary-but-insufficient control                                 | What bypasses it                                                                                                                                        |
| ------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| B1      | `REVOKE EXECUTE ON set_config` (the C1 control)                        | the `SET` SQL command (sets USERSET GUCs without function EXECUTE); `RESET ROLE` (escapes the `SET LOCAL ROLE` frame back to the privileged login role) |
| B2      | `SET LOCAL`/`set_config(…,true)` transaction-local (assumed pool-safe) | a SESSION-level `SET` survives `client.release()` — no `DISCARD/RESET ALL`                                                                              |
| B3      | routine audit gates on `has_function_privilege(EXECUTE)` (`:1100`)     | a SECURITY DEFINER function wired as a TRIGGER fires via DML, needs no EXECUTE → excluded; no `pg_trigger` scan                                         |

The through-line: the security model trusts session GUCs (`kovo.principal`, `kovo.role`) and a `SET LOCAL ROLE
kovo_reader` frame, but BOTH are app-settable/escapable, and the ACTUAL sole control is the managed-handle
single-statement confinement (`managed-db.ts:1078-1083`, rejects `db.exec`, requires one parameterized statement) —
which is undocumented as the security boundary and is defeated by any adjacent path that can issue a utility statement.

Baseline: default Postgres/PGlite at `origin/main`. B1 (both `SET`-forge and `RESET ROLE`-escape) and the set_config
function-path being correctly blocked were self-verified first-hand on raw PGlite; B2/B3 roots self-verified in source;

## Issues

- [ ] **B1 — App SQL forges `kovo.principal` (impersonate any owner) and `kovo.role='admin'/'system'` (escalate into the crossOwnerRead/system RLS policy → read/write ALL owners) via the plain `SET` command, and escapes the `SET LOCAL ROLE kovo_reader` frame via `RESET ROLE`. The `set_config` EXECUTE REVOKE (followup-9 C1) is a necessary-but-insufficient control.** (HIGH, framework/security, architectural; converged from `pg-auth-brand-crossowner` PGAUTH-GUC-1 + `pg-provision-leastpriv-turnkey` PG14-GUC-FORGE + `pg-principal-role-forge` PRF-1 + `pg-egress-wire` EW-1; self-verified first-hand)
  - Observed (raw-PGlite, framework flow — login role sets `kovo.principal`/`kovo.role` then `SET LOCAL ROLE kovo_reader`, then app SQL runs): as `kovo_reader`, `SET kovo.role = 'admin'` SUCCEEDS → `current_setting('kovo.role')='admin'` → the reader reads BOTH owners via the admin policy (`postgres-runtime.ts:2431`) — cross-owner leak. `SET kovo.role='system'` → the system policy (`:2410-2411`) → cross-owner read+write. `SET kovo.principal='bob'` → impersonation (alice's session reads bob's rows). Separately, `RESET ROLE` after `SET LOCAL ROLE kovo_reader` returns to the privileged session/login role, which HAS `set_config` EXECUTE → forge via the function too.
  - Root cause (self-verified): `kovo.principal`/`kovo.role` are plain USERSET custom GUCs (`postgres-runtime.ts:2308/2360/2410/2431` read them via `current_setting`) with NO `GRANT SET ON PARAMETER`/restrictive context, so ANY role can set them with the `SET` command — which does NOT require the `set_config()` function EXECUTE that C1 REVOKEd (`:668`). And `SET LOCAL ROLE` is not a confinement: `RESET ROLE`/`SET ROLE <login>` returns to the session role. The ONLY actual barrier is the managed-handle single-statement confinement (`managed-db.ts:1078-1083`), which is a different mechanism, undocumented as the boundary. `has_function_privilege('kovo_reader','set_config','EXECUTE')=false` confirms the function path is (correctly) blocked — it is the `SET` command + `RESET ROLE` paths that are open. First-hand probes: `scratchpad/set-guc-forge.mjs` (SET forge), `scratchpad/reset-role-escape.mjs` (RESET ROLE → set_config).
  - Why it matters: the ENTIRE owner-scoping guarantee and the crossOwnerRead admin escape reduce to "no app-reachable path can issue a `SET`/`RESET ROLE`." That is the load-bearing invariant, and it is neither the documented control (the plan says `set_config` REVOKE is) nor airtight — any path that runs a utility statement (the node-pg simple/multi-statement protocol, a durable task, a webhook, a raw-SQL escape, a future Drizzle method emitting `SET`) forges cross-owner access + admin/system escalation on a green build. It directly violates followup-9's C7 on followup-9's own C1 fix.
  - Reachability (severity hinge): the DEFAULT managed query path blocks a bare `SET` (`managed-db.ts:1078-1083` rejects `db.exec`, requires one parameterized statement); the reproductions reach it via the node-pg simple protocol (PRF-1) and confirm the mechanism. HIGH because the whole model rests on the confinement being complete and acknowledged, and multiple reachable paths defeat it.
  - Acceptance: (1) crossOwnerRead/system elevation must use an UNASSUMEABLE ROLE (`SET LOCAL ROLE kovo_admin`, which followup-9 B2 already makes app-unassumeable) rather than a forgeable GUC; (2) owner scoping's `kovo.principal` dependency must be protected by an ACKNOWLEDGED sole door — prove (lint+test) that NO managed/task/webhook/escape path can execute `SET`/`RESET ROLE`/`SET ROLE`/multi-statement touching `kovo.*` or the role frame; ideally set `kovo.principal` via a mechanism app SQL cannot override (e.g. a `GRANT SET ON PARAMETER` restriction where supported, or a per-statement bind that RLS reads from a non-settable source). A paranoid test: an app-reachable `SET kovo.role='admin'` / `RESET ROLE` must NOT widen the read set.

- [ ] **B2 — Pooled Postgres connections are released without `DISCARD ALL`/`RESET ALL`, so a SESSION-level `kovo.role`/`kovo.principal` (or role frame) set on a connection persists to the NEXT request — a different tenant — on connection reuse.** (MED, framework/security; `pg-auth-brand-crossowner` PGAUTH-POOL-2; reproduced + root self-verified)
  - Observed: a session-level `SET kovo.role='admin'` (no `LOCAL`) or a leaked role is not cleared when the connection returns to the pool; the next borrower inherits it, reading/writing as the wrong role/principal across tenants.
  - Root cause (self-verified): `grep 'DISCARD|RESET ALL|RESET ROLE'` in `postgres-runtime.ts`/`managed-db.ts` = 0; `client.release()` (`postgres-runtime.ts:1426`) returns the connection with no reset. The framework relies on `SET LOCAL`/`set_config(…,true)` being transaction-local (reset at COMMIT), which is true for the framework's OWN calls — but B1's session-level `SET` (without `LOCAL`) survives `release()` and applies to the next request. So B2 is the cross-tenant PERSISTENCE vector for B1's forge, and a defense-in-depth gap independent of it.
  - Why it matters: even a single leaked session GUC (via B1, or a future bug, or an app's own `SET`) becomes cross-tenant because the pool does not scrub connections; `DISCARD ALL` on release is the standard RLS-on-a-pool hygiene and is absent.
  - Repro evidence: `postgres-runtime.ts:1426` `client.release()` with no `DISCARD`; a session `SET kovo.role='admin'` persists across borrow.
  - Acceptance: every pooled connection is reset on release (`DISCARD ALL`, or at minimum `RESET ROLE; RESET ALL`) OR the framework proves all `kovo.*`/role state is strictly transaction-local and app SQL cannot set a session-level GUC (pairs with B1). A test: a session GUC set on one borrow is absent on the next.

- [ ] **B3 — SECURITY DEFINER TRIGGER functions escape the routine closure audit: a definer trigger on an app-writable table fires on a normal writer INSERT, runs with its definer's privileges, and exfiltrates cross-owner data (or performs cross-owner writes) on a GREEN `checkPostgresAppDbPosture`.** (HIGH, framework/security, architectural; `pg-fuzzer-completeness` PGFC-1; reproduced against the real gate + root self-verified)
  - Observed: a `SECURITY DEFINER` function owned by a non-superuser `app_admin` (that owns a side `billing_secrets` table no app role can reach) is wired `AFTER INSERT ON <owner_table> … EXECUTE FUNCTION exfil_trg()`. `checkPostgresAppDbPosture` → `{ok:true, issues:[]}`, `has_function_privilege(kovo_writer, exfil_trg, 'EXECUTE')=false`, `appRoleCanReachBilling=false`. A normal writer transaction (`SET LOCAL ROLE kovo_writer; SET LOCAL kovo.principal='attacker'; INSERT …`) FIRES the trigger and returns `EXFIL:PAN-4111-1111` via the trigger's `RAISE` — a value no app role can read directly. A superuser/BYPASSRLS definer variant additionally achieves cross-owner INSERT into the FORCE-RLS owner table.
  - Root cause (self-verified): `auditPostgresReachableRoutines` (`postgres-runtime.ts:1080-1118`) selects `pg_proc WHERE prosecdef=true` across all non-system schemas but gates on `AND has_function_privilege(r.oid, p.oid, 'EXECUTE')` (`:1100`). Trigger functions fire via DML and NEVER require the invoking role to hold EXECUTE (and cannot be called directly), so a definer trigger function has `can_execute=false` for every app role and is dropped from the audited set. There is NO `pg_trigger`/`tgfoid` scan anywhere (`grep` = 0). This is the C7 meta-bug: the EXECUTE-reachability gate is a necessary-but-insufficient proxy for a routine's actual reachability (a trigger is reachable via a WRITE the app role legitimately performs).
  - Why it matters: SECURITY DEFINER triggers are the canonical RLS-bypass vector, and auditing definer routines was followup-8's own hardening; wiring the identical function as a trigger reopens the exact class one indirection over. Triggers (audit/denorm/updated_at) are an ordinary migration pattern the closure audit promises to vet. It falsifies the DEC-F fuzzer-completeness claim — the cross-product covers `definer-function × EXECUTE-grant` but NOT `definer-function-as-TRIGGER`.
  - Repro evidence: `/Users/mini/kovo-dogfood-round14/pg-fuzzer-completeness`; `checkPostgresAppDbPosture` green while a writer INSERT exfiltrates via the definer trigger. Root: `postgres-runtime.ts:1080-1118` (esp. `:1100`); no `pg_trigger` scan.
  - Acceptance: the audit enumerates `pg_trigger` and, for every trigger on an app-role-writable/readable table, treats its trigger function under the SAME definer/RLS-capability rule as an EXECUTE-reachable routine — a `SECURITY DEFINER` trigger function reachable via app-role DML is refused unless vetted. Add the trigger axis to the DEC-F fuzzer cross-product. A test: a definer trigger on an owner table makes `kovo db check` REFUSE.
